2. Created template/prompts/prompt_metadata.json¶

Comprehensive metadata file containing:

  • Prompt Definitions: Full documentation for each prompt

    • Type specifications

    • Choice descriptions with rationale

    • Default values with conditional logic

    • Validation rules

    • Dependencies between prompts

    • Affected systems/files

  • Compatibility Matrix: Documents incompatible combinations

    • Django vs. Vite/Next.js conflicts

    • Next.js vs. other meta-frameworks

    • shadcn-ui requirements (React + Tailwind)

    • State management framework requirements

  • Tool Dependencies: Maps languages to required CLI tools

    • Python → python3, pip

    • Rust → cargo, rustc

    • Node → node, npm

    • Go → go

    • Java → java, javac

    • C# → dotnet

3. Created template/prompts/copier-answers-schema.yml¶

JSON Schema for validating copier answers:

  • Schema Definitions: Type definitions for all prompts

  • Enum Validations: Valid choices for each prompt

  • Array Validations: Constraints for multiselect prompts (uniqueItems, minItems)

  • Pattern Validations: Regex patterns for project_name, project_slug, package_name

  • Required Fields: Identifies mandatory prompts

  • Conditional Fields: Documents when-clause dependencies

A2: Validation Framework ✅ VALIDATORS COMPLETE¶

Implemented Validators¶

1. Directory Structure¶

Created template/hooks/validators/ with:

  • __init__.py - Module initialization and exports

  • language_validator.py - Backend language validation

  • frontend_validator.py - Frontend framework validation

  • tool_validator.py - Development tool validation

  • structure_validator.py - Repository structure validation

2. language_validator.py¶

Features:

  • validate_backend_languages() - Ensures at least one language selected

  • validate_language_framework_compatibility() - Checks framework conflicts

    • Django incompatibility with Vite/Next.js

    • Rust framework/runtime compatibility (e.g., Axum requires Tokio)

  • validate_package_managers() - Verifies package managers are installed

  • Tool availability checks using shutil.which()

Data Structures:

  • LANGUAGE_TOOLS: Required tools per language

  • FRAMEWORK_COMPATIBILITY: Framework requirements and conflicts

  • Package manager command mappings

3. frontend_validator.py¶

Features:

  • validate_frontend_framework() - Ensures frontend config consistency

  • validate_component_library_compatibility() - Checks library requirements

    • shadcn-ui requires React + Tailwind

    • MUI/Chakra/Mantine require React

    • Ant Design supports React + Vue

  • validate_state_management_compatibility() - Framework-specific state mgmt

    • Zustand/Redux/Jotai → React only

    • Pinia/Vuex → Vue only

  • validate_meta_framework_compatibility() - Prevents meta-framework conflicts

    • Next.js/Remix/Gatsby mutual exclusivity

    • Nuxt/Quasar/VitePress conflicts

Data Structures:

  • COMPONENT_LIBRARY_REQUIREMENTS: Library dependencies

  • STATE_MANAGEMENT_COMPATIBILITY: State library framework requirements

  • META_FRAMEWORK_CONFLICTS: Mutually exclusive frameworks

4. tool_validator.py¶

Features:

  • validate_language_tools() - Checks language-specific CLIs

  • validate_package_managers() - Verifies pm availability (uv, pnpm, etc.)

  • validate_build_tools() - Warns about missing build tools (non-blocking)

  • validate_monorepo_tools() - Warns about missing monorepo tools

  • validate_ide_tools() - Informational checks for IDE CLI commands

  • Returns categorized results: errors, warnings, info

Data Structures:

  • REQUIRED_TOOLS: Language → required commands

  • PACKAGE_MANAGER_COMMANDS: PM name → CLI command

  • BUILD_TOOL_COMMANDS: Build tool → CLI command

  • MONOREPO_TOOL_COMMANDS: Monorepo tool → CLI command

  • IDE_COMMANDS: IDE → CLI command

5. structure_validator.py¶

Features:

  • validate_structure_compatibility() - Ensures structure matches project_layout

    • apps-packages/multi-workspace require monorepo

    • flat/src-layout not recommended for monorepos

  • validate_structure_language_compatibility() - Warns about polyglot issues

    • Multiple languages with flat/src-layout → suggest apps-packages

  • validate_structure_tools() - Recommends appropriate monorepo tools

    • apps-packages → Turborepo/Nx

    • multi-workspace → pnpm/yarn workspaces

  • validate_structure_service_count() - Checks if structure fits service count

    • Warns if too many services for simple structures

    • Warns if too few services for complex structures

Data Structures:

  • STRUCTURE_REQUIREMENTS: Structure-specific recommendations and constraints

2. Structure Definitions¶

Flat Structure:

structure:
  - src/
  - tests/
  - docs/
  - scripts/

Src-Layout Structure:

structure:
  - src/
    - {{ package_name }}/
      - __init__.py
      - main.py
  - tests/
  - docs/

Apps-Packages Structure:

structure:
  - apps/
    {% if 'python' in backend_languages %}
    - api/
    {% endif %}
    {% if frontend_framework != 'none' %}
    - web/
    {% endif %}
    {% if cli_module == 'enabled' %}
    - cli/
    {% endif %}
  - packages/
    {% if shared_logic == 'enabled' %}
    - shared/
    {% endif %}
    - config/
  - tooling/

Multi-Workspace Structure:

structure:
  - backend/
    - services/
    - package.json
  - frontend/
    - apps/
    - package.json
  - shared/
    - libs/
    - package.json
  - pnpm-workspace.yaml

Domain-Driven Structure:

structure:
  - domains/
    - user/
    - product/
    - order/
  - shared/
    - kernel/
    - infrastructure/
