# Configuration Edge Cases & Conflict Resolution

```{admonition} Purpose
:class: note

This guide documents known edge cases, configuration conflicts, and resolution strategies in the Riso template framework. It complements the [Configuration Decision Guide](configuration-decision-guide.md) by focusing on validation rules, error handling, and complex configuration scenarios.
```

## Overview

The Riso template uses a multi-layered validation framework to detect and report configuration conflicts:

- **Error-level issues** — Blocking incompatibilities that prevent project generation
- **Warning-level issues** — Non-blocking incompatibilities with workarounds or performance implications
- **Info-level notices** — Configuration notes and behavioral changes
- **Recommendations** — Suggestions for optimal tool combinations

## Validation Architecture

### Validation Flow

```mermaid
graph TD
    A[User Answers] --> B[pre_gen_project.py]
    B --> C{Load Context}
    C --> D[SaaS Starter Validation]
    C --> E[Tooling Validation]
    C --> F[Module Validators]

    D --> G{Check Errors}
    F --> G

    G -->|Has Errors| H[Exit with Error Message]
    G -->|No Errors| I{Check Warnings}

    I -->|Has Warnings| J[Display Warnings & Continue]
    I -->|No Warnings| K[Continue Generation]
    J --> K

    F --> L[Frontend Validator]
    F --> M[Build Tool Validator]
    F --> N[Monorepo Tool Validator]
    F --> O[Structure Validator]
    F --> P[Language Validator]
    F --> Q[Tool Validator]
```

### Validator Locations

| Validator | Path | Responsibility |
|-----------|------|----------------|
| Pre-generation hook | `template/hooks/pre_gen_project.py` | Orchestrates all validations |
| SaaS validator | Embedded in pre-gen hook | SaaS Starter configuration rules |
| Frontend validator | `template/hooks/validators/frontend_validator.py` | Frontend framework compatibility |
| Build tool validator | `template/hooks/validators/build_tool_validator.py` | Build tool compatibility |
| Monorepo validator | `template/hooks/validators/monorepo_tool_validator.py` | Monorepo tool compatibility |
| Structure validator | `template/hooks/validators/structure_validator.py` | Repository structure validation |
| Language validator | `template/hooks/validators/language_validator.py` | Backend language validation |
| Tool validator | `template/hooks/validators/tool_validator.py` | Tool availability checks |

## SaaS Starter Edge Cases

### Error-Level Incompatibilities

#### Neon Database + Supabase Storage

**Conflict:** Cannot use Neon database with Supabase Storage.

**Reason:** Supabase Storage requires Supabase's authentication and database layer for Row-Level Security (RLS) policies.

**Resolution Options:**

1. **Full Supabase Stack** (recommended for integrated platforms):
   ```yaml
   saas_database: supabase
   saas_storage: supabase-storage
   saas_auth: clerk  # or authjs
   ```

2. **Neon + R2 Stack** (recommended for edge deployment):
   ```yaml
   saas_database: neon
   saas_storage: r2
   saas_auth: clerk  # or authjs
   ```

**Detection:** Pre-generation hook checks if both `neon` database and `supabase-storage` are selected.

**Error Message:**
```
Cannot use Neon database with Supabase Storage. Choose either:
  1. Full Supabase (database + storage)
  2. Neon database + Cloudflare R2 storage
```

### Warning-Level Incompatibilities

#### Cloudflare Workers + Prisma ORM

**Conflict:** Prisma requires TCP connections, which Cloudflare Workers don't support.

**Reason:** Cloudflare Workers run in V8 isolates without TCP socket access. Prisma uses direct TCP connections to databases.

**Workaround:** Use Prisma Data Proxy (adds latency ~50-150ms and monthly cost).

**Resolution Options:**

1. **Switch to Drizzle ORM** (recommended):
   ```yaml
   saas_hosting: cloudflare
   saas_orm: drizzle  # Edge-compatible
   ```

2. **Use Prisma Data Proxy**:
   - Enable Prisma Data Proxy in your Prisma Cloud account
   - Update connection string: `prisma://proxy.prisma-data.com/?api_key=...`
   - Accept latency trade-off (~50-150ms per query)

3. **Switch to Vercel hosting**:
   ```yaml
   saas_hosting: vercel  # Supports native Prisma
   saas_orm: prisma
   ```

**Performance Impact:**
- Direct connection (Vercel): ~1-5ms query latency
- Prisma Data Proxy (Cloudflare): ~50-150ms query latency
- Drizzle on Edge (Cloudflare): ~1-10ms query latency

**Detection:** Pre-generation hook checks if both `cloudflare` hosting and `prisma` ORM are selected.

**Warning Message:**
```
⚠️  Prisma requires TCP connections, which Cloudflare Workers don't support.
You'll need to use Prisma Data Proxy (adds latency and cost).
Recommendation: Use Drizzle ORM for better edge compatibility.
```

#### Vercel Hosting + Cloudflare R2 Storage

**Conflict:** Cloudflare R2 works with Vercel but incurs egress bandwidth charges.

**Reason:** Cross-cloud egress from Cloudflare R2 to Vercel functions incurs Cloudflare's bandwidth fees.

**Cost Implications:**
- Cloudflare R2 egress to Vercel: $0.09/GB (after 10GB free monthly)
- Vercel Blob storage: $0.15/GB stored + $0.30/GB transferred (included in Pro plan)

**Resolution Options:**

1. **Accept egress charges** (best for low traffic):
   - Good if serving <100GB/month
   - Monitor bandwidth usage
   - Set up Cloudflare Analytics alerts

2. **Switch to Vercel Blob** (best for Vercel-native deployments):
   ```yaml
   saas_hosting: vercel
   saas_storage: vercel-blob  # Note: Not currently a template option
   ```

3. **Switch to Cloudflare Pages** (best for edge deployment):
   ```yaml
   saas_hosting: cloudflare
   saas_storage: r2  # No egress fees within Cloudflare
   ```

**Detection:** Pre-generation hook checks if both `vercel` hosting and `r2` storage are selected.

**Warning Message:**
```
⚠️  Cloudflare R2 works with Vercel but egress bandwidth charges apply.
```

### Info-Level Notices

#### Supabase Database + Clerk Auth

**Notice:** Supabase Auth will be disabled when using Clerk.

**Reason:** Using two authentication systems creates conflicts. Clerk provides more features (organizations, passkeys) than Supabase Auth.

**Behavioral Changes:**
- Supabase Auth APIs disabled
- Auth tables/schemas not created
- Row-Level Security (RLS) must use Clerk JWTs
- Supabase Storage RLS must reference Clerk user IDs

**Integration Pattern:**
```typescript
// Use Clerk JWTs with Supabase RLS
import { auth } from '@clerk/nextjs';
import { createClient } from '@supabase/supabase-js';

const supabase = createClient(
  process.env.SUPABASE_URL!,
  process.env.SUPABASE_ANON_KEY!,
  {
    global: {
      headers: {
        Authorization: `Bearer ${await auth().getToken({ template: 'supabase' })}`,
      },
    },
  }
);
```

**Detection:** Pre-generation hook checks if both `supabase` database and `clerk` auth are selected.

**Info Message:**
```
ℹ️  You've selected both Supabase and Clerk.
Supabase Auth will be disabled in favor of Clerk.
```

## Frontend Framework Edge Cases

### Component Library Incompatibilities

#### shadcn/ui Requirements

**Requirements:**
- Frontend framework: React only
- CSS framework: Tailwind CSS only

**Conflict Scenarios:**

1. **shadcn/ui + Vue**:
   ```yaml
   # ❌ Invalid
   frontend_framework: vue
   component_library: shadcn-ui
   ```

   **Error:** "shadcn/ui requires React framework, but vue was selected. shadcn/ui is built specifically for React with Tailwind CSS"

   **Resolution:** Use Headless UI or Radix Vue instead:
   ```yaml
   frontend_framework: vue
   component_library: headless-ui  # or none
   ```

2. **shadcn/ui without Tailwind**:
   ```yaml
   # ❌ Invalid
   frontend_framework: react
   component_library: shadcn-ui
   css_framework: styled-components
   ```

   **Error:** "shadcn/ui requires Tailwind CSS framework, but styled-components was selected"

   **Resolution:** Add Tailwind CSS:
   ```yaml
   frontend_framework: react
   component_library: shadcn-ui
   css_framework: tailwind
   ```

#### Component Library Compatibility Matrix

| Component Library | Compatible Frameworks | Required CSS | Notes |
|-------------------|----------------------|--------------|-------|
| shadcn/ui | React | Tailwind | Most popular React+Tailwind combo |
| Material-UI (MUI) | React | None (CSS-in-JS) | Complete design system |
| Ant Design | React, Vue | None (CSS-in-JS) | Enterprise-focused |
| Chakra UI | React | None (CSS-in-JS) | Accessible components |
| Mantine | React | None (CSS-in-JS) | Full-featured with hooks |
| NextUI | React | None (built-in) | Modern, fast |
| Headless UI | React, Vue | Any | Unstyled, accessible |
| Tremor | React | Tailwind | Analytics/dashboard focus |
| Preline | Any | Tailwind | Tailwind plugin |

### State Management Incompatibilities

#### React State Management with Vue

**Conflict:** Zustand, Redux, Jotai, Valtio only work with React.

**Error Scenario:**
```yaml
# ❌ Invalid
frontend_framework: vue
react_state_management: zustand
```

**Error:** "zustand is not compatible with Vue. It's designed for react."

**Resolution:** Use Vue-specific state management:
```yaml
frontend_framework: vue
vue_state_management: pinia  # Modern Vue 3 state management
```

#### State Management Compatibility Matrix

| State Library | Compatible Frameworks | Bundle Size | Learning Curve | Best For |
|---------------|----------------------|-------------|----------------|----------|
| Zustand | React | ~1KB | Easy | Simple global state |
| Redux Toolkit | React | ~12KB | Moderate | Complex state, DevTools |
| Jotai | React | ~3KB | Easy | Atomic state, concurrent mode |
| Valtio | React | ~3KB | Easy | Proxy-based state |
| MobX | React | ~16KB | Moderate | Observable state |
| Context API | React | 0KB (built-in) | Easy | Small apps, theme/auth |
| Pinia | Vue | ~2KB | Easy | Vue 3 (official) |
| Vuex | Vue | ~10KB | Moderate | Vue 2/3 (legacy) |

### Meta-Framework Conflicts

#### Multiple React Meta-Frameworks

**Conflict:** Next.js, Remix, Gatsby, Astro, TanStack Router are mutually exclusive.

**Error Scenario:**
```yaml
# ❌ Invalid
frontend_framework: react
react_framework: nextjs
web_app_engine: remix
```

**Error:** "nextjs and remix are mutually exclusive. Choose one meta-framework."

**Resolution:** Choose one based on use case:

| Framework | Best For | Routing | Data Loading | Deployment |
|-----------|----------|---------|--------------|------------|
| Next.js | Full-stack apps, SEO | File-based | Server Components | Vercel, Cloudflare |
| Remix | Complex forms, mutations | Nested routes | Loaders/actions | Cloudflare, Fly.io |
| Gatsby | Static sites, blogs | File-based | GraphQL | Netlify, Vercel |
| Astro | Content sites | File-based | Islands | Any static host |
| TanStack Router | SPAs | Type-safe routes | Loaders | Client-side |

## Build Tool Edge Cases

### Turbopack Exclusivity

**Limitation:** Turbopack only works with Next.js.

**Conflict Scenario:**
```yaml
# ❌ Invalid
frontend_framework: react
react_framework: vite
build_tool: turbopack
```

**Error:** "Build tool 'turbopack' is not compatible with 'vite' framework. Compatible frameworks: react"

**Additional Context:** Turbopack is Next.js's built-in bundler, not a standalone tool.

**Resolution:** Use framework-appropriate build tools:

```yaml
# Next.js → Turbopack or Webpack
frontend_framework: react
react_framework: nextjs
build_tool: turbopack

# Remix → Vite or esbuild
frontend_framework: react
react_framework: remix
build_tool: vite

# Vanilla React → Vite, Webpack, Rspack
frontend_framework: react
react_framework: none
build_tool: vite
```

### Build Tool Compatibility Matrix

| Build Tool | Frameworks | Strengths | Weaknesses | Typical Build Time |
|------------|-----------|-----------|------------|-------------------|
| Vite | React, Vue, Svelte, Solid | Fast HMR, modern defaults | Limited legacy support | 1-5s (dev), 10-30s (prod) |
| Webpack | All | Ecosystem, plugins | Slow, complex config | 5-20s (dev), 30-120s (prod) |
| esbuild | React, Solid | Extremely fast | Limited plugins | <1s (dev), 5-10s (prod) |
| Rollup | React, Vue, Svelte | Library builds, tree-shaking | Slow for apps | 10-30s (prod) |
| Rspack | React, Vue, Svelte | Fast, Webpack-compatible | Newer, fewer plugins | 2-8s (dev), 15-45s (prod) |
| Turbopack | Next.js only | Fastest for Next.js | Next.js exclusive | 1-3s (dev), 10-25s (prod) |
| Parcel | React, Vue, Svelte | Zero config | Less control | 3-10s (dev), 20-60s (prod) |

### esbuild Plugin Limitations

**Warning:** esbuild has limited plugin support for Vue and Svelte.

**Affected Scenarios:**
```yaml
# ⚠️  Warning
frontend_framework: vue
build_tool: esbuild
```

**Warning:** "esbuild has limited plugin support for vue. Consider using Vite or Rollup instead."

**Workarounds:**
1. **Switch to Vite** (recommended): Vite uses esbuild internally but adds Vue plugin support
2. **Use Rollup**: Slower but full Vue plugin ecosystem
3. **Accept limitations**: esbuild can compile Vue SFCs but lacks advanced features

**Missing esbuild Features:**
- Hot Module Replacement (HMR) for Vue
- Vue Devtools integration
- Advanced template compilation optimizations
- CSS scoping edge cases

## Monorepo Tool Edge Cases

### Package Manager Incompatibility

#### Lerna + Bun

**Conflict:** Lerna doesn't support Bun package manager yet.

**Error Scenario:**
```yaml
# ❌ Invalid
project_layout: monorepo
monorepo_tool: lerna
node_package_manager: bun
```

**Error:** "Monorepo tool 'lerna' is not compatible with package manager 'bun'. Compatible managers: pnpm, npm, yarn"

**Resolution Options:**

1. **Switch to Turborepo or Nx** (supports Bun):
   ```yaml
   monorepo_tool: turborepo  # or nx
   node_package_manager: bun
   ```

2. **Switch to pnpm** (best monorepo experience):
   ```yaml
   monorepo_tool: lerna
   node_package_manager: pnpm
   ```

#### Package Manager Compatibility Matrix

| Monorepo Tool | pnpm | npm | Yarn Classic | Yarn Berry | Bun |
|---------------|------|-----|--------------|------------|-----|
| Turborepo | ✅ | ✅ | ✅ | ✅ | ✅ |
| Nx | ✅ | ✅ | ✅ | ✅ | ✅ |
| Lerna | ✅ | ✅ | ✅ | ⚠️ Limited | ❌ |
| Moon | ✅ | ✅ | ✅ | ✅ | ✅ |
| Bazel | N/A (language-agnostic) | N/A | N/A | N/A | N/A |

### Repository Structure Mismatches

#### Flat Structure + Monorepo Tool

**Warning:** Flat structure not ideal for monorepo tools.

**Scenario:**
```yaml
# ⚠️  Warning
project_layout: monorepo
repository_structure: flat
monorepo_tool: turborepo
```

**Warning:** "Repository structure 'flat' is not ideal for monorepo tool 'turborepo'. Consider using 'apps-packages' or 'multi-workspace' structure."

**Recommended Structures:**

| Structure | Layout | Best For | Example |
|-----------|--------|----------|---------|
| `apps-packages` | `apps/`, `packages/` | Mixed apps + shared libs | Next.js app + React components |
| `multi-workspace` | Multiple workspace roots | Independent services | API + Web + Mobile |
| `domain-driven` | Domain-based folders | Enterprise apps | `auth/`, `billing/`, `analytics/` |

**Resolution:**
```yaml
# Recommended for monorepo tools
project_layout: monorepo
repository_structure: apps-packages
monorepo_tool: turborepo
```

### Language Support Limitations

#### Turborepo with Python/Rust

**Warning:** Turborepo is optimized for Node.js but can work with other languages.

**Scenario:**
```yaml
# ⚠️  Warning
monorepo_tool: turborepo
backend_languages: [python, rust]
```

**Warning:** "Monorepo tool 'turborepo' is optimized for Node.js but your project uses: python, rust. Consider using 'moon' or 'bazel' for better multi-language support."

**Language Support Matrix:**

| Tool | Node.js | Python | Rust | Go | Java | Multi-Language Quality |
|------|---------|--------|------|----|----- |-----------------------|
| Turborepo | ✅ Excellent | ⚠️ Basic | ⚠️ Basic | ⚠️ Basic | ❌ | Poor (Node-focused) |
| Nx | ✅ Excellent | ✅ Good | ✅ Good | ✅ Good | ⚠️ Basic | Good (plugins available) |
| Lerna | ✅ Excellent | ❌ | ❌ | ❌ | ❌ | None (Node-only) |
| Moon | ✅ Excellent | ✅ Excellent | ✅ Excellent | ✅ Excellent | ⚠️ Basic | Excellent (language-agnostic) |
| Bazel | ✅ Good | ✅ Excellent | ✅ Excellent | ✅ Excellent | ✅ Excellent | Excellent (enterprise-grade) |

**Resolution for Multi-Language Projects:**
```yaml
# Best for Python + Rust + Node.js
monorepo_tool: moon
backend_languages: [python, rust, node]
```

### Project Scale Mismatches

#### Bazel for Small Projects

**Warning:** Bazel is designed for very large monorepos (100+ packages).

**Scenario:**
```yaml
# ⚠️  Warning
monorepo_tool: bazel
repository_structure: apps-packages  # Estimated 5 packages
```

**Warning:** "Bazel is designed for very large monorepos (100+ packages). For smaller projects, consider 'turborepo' or 'nx' instead."

**Scale Recommendations:**

| Tool | Ideal Package Count | Setup Complexity | Build Speed | Remote Caching |
|------|-------------------|------------------|-------------|----------------|
| Turborepo | 2-50 | Low | Fast | ✅ (Vercel/custom) |
| Nx | 5-200 | Medium | Fast | ✅ (Nx Cloud/custom) |
| Lerna | 3-30 | Low | Medium | ❌ (use with Nx) |
| Moon | 3-100 | Medium | Fast | ✅ (built-in) |
| Bazel | 20-10000+ | High | Very Fast | ✅ (enterprise-grade) |

## Repository Structure Edge Cases

### Structure-Layout Conflicts

#### Monorepo Layout with Single-Package Structure

**Error:** Certain structures require monorepo layout.

**Scenario:**
```yaml
# ❌ Invalid
project_layout: single-package
repository_structure: apps-packages
```

**Error:** "Repository structure 'apps-packages' requires project_layout='monorepo', but 'single-package' was selected. Change project_layout to 'monorepo'."

**Resolution:**
```yaml
project_layout: monorepo
repository_structure: apps-packages
```

#### Structure-Layout Compatibility

| Structure | Requires Monorepo? | Best Project Layout | Example Use Case |
|-----------|-------------------|---------------------|------------------|
| `flat` | No | `single-package` | Simple CLI tool |
| `src-layout` | No | `single-package` | Python package following PEP 517 |
| `apps-packages` | Yes | `monorepo` | Full-stack app with shared libs |
| `multi-workspace` | Yes | `monorepo` | Multiple independent apps |
| `domain-driven` | No (but recommended) | Either | Large app by business domains |

### Multiple Languages with Simple Structures

**Warning:** Multiple backend languages with flat/src-layout structures.

**Scenario:**
```yaml
# ⚠️  Warning
repository_structure: src-layout
backend_languages: [python, rust, node]
```

**Warning:** "Multiple backend languages (python, rust, node) selected with 'src-layout' structure. Consider 'apps-packages' or 'multi-workspace' for better polyglot project organization."

**Resolution for Polyglot Projects:**
```yaml
repository_structure: apps-packages
backend_languages: [python, rust, node]
monorepo_tool: moon  # Best multi-language support
```

**Directory Structure Example:**
```
project/
├── apps/
│   ├── api-python/      # FastAPI service
│   ├── api-node/        # Fastify service
│   └── cli-rust/        # Rust CLI tool
├── packages/
│   ├── shared-types/    # TypeScript type definitions
│   └── shared-utils/    # Shared utilities
└── libs/
    ├── python-common/   # Python shared code
    └── rust-common/     # Rust shared code
```

### Service Count Mismatches

#### Monorepo Structure with Single Service

**Warning:** Using monorepo structure for simple projects.

**Scenario:**
```yaml
# ⚠️  Warning
repository_structure: apps-packages
cli_module: disabled
backend_languages: []
frontend_framework: none
docs_site: none
```

**Warning:** "Repository structure 'apps-packages' is designed for at least 2 services, but only 1 detected. Consider 'src-layout' for simpler projects."

**Service Count Estimation:**
- CLI module enabled: +1
- Each backend language: +1
- Frontend framework (not none): +1
- Docs site (not none): +1

**Resolution for Simple Projects:**
```yaml
repository_structure: src-layout
```

## Tool Availability Edge Cases

### Missing Required Tools

#### Python Project without uv

**Error:** Required tools not found in PATH.

**Scenario:** Running copier without uv installed.

**Error:** "Python package manager 'uv' (uv) not found in PATH. Install it or select a different package manager."

**Resolution:**

1. **Install uv** (recommended):
   ```bash
   curl -LsSf https://astral.sh/uv/install.sh | sh
   ```

2. **Use different package manager**:
   ```bash
   copier copy gh:yourusername/riso my-project \
     --data python_package_manager=poetry
   ```

#### Node.js Project without pnpm

**Error:** Node.js package manager not found.

**Error:** "Node.js package manager 'pnpm' (pnpm) not found. Install it with: npm install -g pnpm"

**Resolution:**

1. **Install pnpm**:
   ```bash
   npm install -g pnpm
   # or with Homebrew
   brew install pnpm
   # or with mise
   mise install pnpm@latest
   ```

2. **Use npm instead**:
   ```bash
   copier copy gh:yourusername/riso my-project \
     --data node_package_manager=npm
   ```

### Missing Optional Tools

#### Build Tools Not Found (Non-Blocking)

**Warning:** Build tool will be installed as project dependency.

**Warning:** "Build tool 'vite' (vite) not found. It will be installed as a project dependency."

**Behavior:** Template continues generation. Build tool installed during `npm install` or `pnpm install`.

#### Monorepo Tools Not Found (Non-Blocking)

**Warning:** Monorepo tool should be installed globally.

**Warning:** "Monorepo tool 'turborepo' (turbo) not found. Install it globally: npm install -g turborepo"

**Resolution (post-generation):**
```bash
npm install -g turborepo
# or use package manager scripts
pnpm run turbo
```

## Complex Valid Configurations

These configurations pass validation but require careful attention:

### Edge-Optimized Stack with Workarounds

```yaml
# Valid but requires Prisma Data Proxy setup
saas_starter_module: enabled
saas_runtime: nextjs-16
saas_hosting: cloudflare
saas_database: neon
saas_orm: prisma  # ⚠️  Requires Data Proxy
saas_auth: clerk

saas_cicd: cloudflare-ci
```

**Setup Requirements:**
1. Enable Prisma Data Proxy in Prisma Cloud
2. Update `DATABASE_URL` to use proxy URL
3. Accept 50-150ms query latency overhead
4. Monitor Data Proxy usage and costs

### Full-Stack Monorepo with Multiple Languages

```yaml
# Complex but fully supported
project_layout: monorepo
repository_structure: apps-packages
monorepo_tool: moon  # Best for multi-language
backend_languages: [python, node, rust]
frontend_framework: react
react_framework: nextjs

node_package_manager: pnpm
```

**Project Structure:**
```
project/
├── apps/
│   ├── web/              # Next.js app (Node.js)
│   ├── api-python/       # FastAPI service (Python)
│   └── cli/              # Rust CLI tool
├── packages/
│   ├── ui/               # Shared React components
│   ├── types/            # TypeScript definitions
│   └── python-common/    # Shared Python utilities
```

### Enterprise SaaS with All Observability

```yaml
# Maximum observability stack
saas_starter_module: enabled
saas_runtime: nextjs-16
saas_hosting: vercel
saas_database: neon
saas_orm: prisma
saas_observability_sentry: true      # Error tracking
```

**Cost Considerations:**

- Datadog: $31/host/month (Pro plan)
- OpenTelemetry: Free (self-hosted) or $0.10/GB (managed)
- Total: ~$60-100/month for observability

**Setup Complexity:** High (requires configuring 3-4 services)

## Adding Custom Validation Rules

### Validator Structure

Create new validator in `template/hooks/validators/`:

```python
"""Custom validator for [feature] configurations."""

from typing import Any, Dict, List


def validate_feature_compatibility(answers: Dict[str, Any]) -> List[str]:
    """Validate [feature] compatibility.

    Args:
        answers: Dictionary of copier answers

    Returns:
        List of error messages (empty if valid)
    """
    errors = []
    feature = answers.get("feature_name")

    # Add validation logic
    if feature == "incompatible_value":
        errors.append("Feature incompatibility message")

    return errors


def validate(answers: Dict[str, Any]) -> List[str]:
    """Run all [feature] validations.

    Args:
        answers: Dictionary of copier answers

    Returns:
        List of all validation error messages
    """
    errors = []
    errors.extend(validate_feature_compatibility(answers))
    return errors
```

### Integrating Validators

Add to `template/hooks/pre_gen_project.py`:

```python
# Import validator
try:
    from hooks.validators.custom_validator import validate as validate_custom
except ModuleNotFoundError:
    validate_custom = lambda answers: []

# Run validation
custom_errors = validate_custom(context)
if custom_errors:
    for error in custom_errors:
        sys.stderr.write(f"  ❌ {error}\n")
    sys.exit(1)
```

### Validation Severity Levels

Use return type to indicate severity:

```python
def validate(answers: Dict[str, Any]) -> Dict[str, List[str]]:
    """Run validations with different severity levels."""
    return {
        "errors": [],      # Blocking (exit 1)
        "warnings": [],    # Non-blocking (display + continue)
        "info": []        # Informational (display)
    }
```

## Troubleshooting Validation Failures

### View Validation Logs

All validation attempts logged to `.riso/toolchain_provisioning.jsonl`:

```bash
# View all validation logs
cat .riso/toolchain_provisioning.jsonl | jq '.'

# Filter by status
cat .riso/toolchain_provisioning.jsonl | jq 'select(.status == "failed")'

# View recent entries
tail -n 20 .riso/toolchain_provisioning.jsonl | jq '.'
```

### Common Error Patterns

#### Error: "Cannot use Neon database with Supabase Storage"

**Cause:** Selecting incompatible SaaS options.

**Fix:**
```bash
# Option 1: Full Supabase
copier copy gh:yourusername/riso my-project \
  --data saas_database=supabase \
  --data saas_storage=supabase-storage

# Option 2: Neon + R2
copier copy gh:yourusername/riso my-project \
  --data saas_database=neon \
  --data saas_storage=r2
```

#### Error: "Build tool 'turbopack' is not compatible with 'vue'"

**Cause:** Selecting framework-exclusive build tool.

**Fix:**
```bash
copier copy gh:yourusername/riso my-project \
  --data frontend_framework=vue \
  --data build_tool=vite
```

#### Error: "Monorepo tool 'lerna' is not compatible with package manager 'bun'"

**Cause:** Unsupported package manager for monorepo tool.

**Fix:**
```bash
# Option 1: Switch to compatible tool
copier copy gh:yourusername/riso my-project \
  --data monorepo_tool=turborepo \
  --data node_package_manager=bun

# Option 2: Switch to compatible package manager
copier copy gh:yourusername/riso my-project \
  --data monorepo_tool=lerna \
  --data node_package_manager=pnpm
```

### Override Validation (Not Recommended)

For testing only, modify `template/hooks/pre_gen_project.py`:

```python
# Temporarily disable specific validator
if context.get("saas_starter_module") == "enabled":
    # Skip validation
    pass
```

**Warning:** This may produce non-functional projects. Only use for debugging validators.
