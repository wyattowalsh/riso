# Validator System

The Riso template uses a comprehensive validation system to ensure project configurations are valid, compatible, and properly tooled before generation. This guide explains the validation pipeline, individual validators, error resolution, and how to extend the system.

## Overview

### Validation Pipeline

The validation system runs during Copier's pre-generation phase via `template/hooks/pre_gen_project.py`. The pipeline executes in this order:

1. **Tool Provisioning** - Attempts to auto-install missing tools (uv, Node.js, pnpm) via mise
2. **Configuration Loading** - Extracts user answers from Copier environment variables
3. **SaaS Starter Validation** - Validates SaaS module configurations (if enabled)
4. **Specialized Validators** - Runs domain-specific validators (languages, frontend, build tools, etc.)
5. **Quality Tool Provisioning** - Ensures Python and Node.js quality tools are available
6. **Failure Reporting** - Aggregates errors and provides actionable guidance

**Key Principles:**

- **Fail Fast**: Renders never proceed in partially configured states
- **Auto-Healing**: Attempts single-pass auto-install before failing
- **Actionable Errors**: Every error includes installation commands or configuration fixes
- **Audit Trail**: All provisioning attempts logged to `.riso/toolchain_provisioning.jsonl`

### Validation Entry Point

```python
# template/hooks/pre_gen_project.py
def main() -> None:
    context = _load_copier_context()

    # SaaS Starter validation (if enabled)
    if context.get("saas_starter_module") == "enabled":
        issues = _validate_saas_starter(context)
        # Block on errors, warn on warnings, inform on notices

    # Tool matrix provisioning
    tool_matrix = [("uv", "0.4", "uv@0.4")]
    if docs_site != "none":
        tool_matrix.extend([
            ("node", "20", "node@20"),
            ("pnpm", "8", "pnpm@8"),
        ])

    # Auto-provision with mise, log attempts
    failures = []
    for tool, version, mise_spec in tool_matrix:
        result = _attempt_install(tool, version, mise_spec)
        _log_attempt(result)
        if result["status"] == "failed":
            failures.append(result)

    # Quality tool checks
    for check in ensure_python_quality_tools():
        # ... provision and log

    # Exit with actionable errors if any failures
    if failures:
        sys.stderr.write("...actionable instructions...")
        sys.exit(1)
```

## Validators

### Language Validator

**Location**: `template/hooks/validators/language_validator.py`

**Purpose**: Validates backend language selections, tool availability, and language-specific framework compatibility.

#### Validation Rules

1. **Language Tool Availability** (`validate_backend_languages`)
   - Checks that required language tools are in PATH
   - Python requires: `python3`
   - Rust requires: `cargo`, `rustc`
   - Node requires: `node`
   - Go requires: `go`
   - Java requires: `java`, `javac`
   - C# requires: `dotnet`

2. **Framework Compatibility** (`validate_language_framework_compatibility`)
   - Validates backend framework matches selected language
   - Checks for incompatibilities with frontend frameworks
   - Example: Django is incompatible with Vite/Next.js/Remix (has its own template system)
   - Example: Axum (Rust) requires Tokio async runtime

3. **Package Manager Availability** (`validate_package_managers`)
   - Python: `uv`, `poetry`, `pdm`, `pip-tools`, `conda`, `pixi`
   - Node: `pnpm`, `npm`, `yarn`, `yarn-berry`, `bun`
   - Validates selected package manager command exists in PATH

#### Example Errors

```bash
# Missing language tool
Python requires 'python3' but it's not found in PATH. Install it before continuing.

# Framework incompatibility
django is incompatible with vite. Django has its own template system

# Runtime mismatch
axum requires tokio async runtime, but async-std was selected

# Missing package manager
Python package manager 'poetry' (poetry) not found in PATH. Install it or select a different package manager.
```

#### Configuration Matrix

| Language | Required Tools | Package Managers | Frameworks |
|----------|----------------|------------------|------------|
| Python | `python3` | uv, poetry, pdm, pip-tools, conda, pixi | Django, Flask, FastAPI |
| Rust | `cargo`, `rustc` | cargo (built-in) | Axum, Actix-web, Rocket, Warp, Poem |
| Node | `node` | pnpm, npm, yarn, yarn-berry, bun | Fastify, Express, Koa, NestJS, Hono |
| Go | `go` | go (built-in) | Gin, Echo, Fiber, Chi, Gorilla |
| Java | `java`, `javac` | maven, gradle | Spring Boot, Quarkus |
| C# | `dotnet` | dotnet (built-in) | ASP.NET Core |

### Frontend Validator

**Location**: `template/hooks/validators/frontend_validator.py`

**Purpose**: Validates frontend framework selections, component library compatibility, and CSS framework requirements.

#### Validation Rules

1. **Frontend Framework Selection** (`validate_frontend_framework`)
   - Ensures component library/CSS framework not selected without frontend
   - Prevents orphaned frontend configurations

2. **Component Library Compatibility** (`validate_component_library_compatibility`)
   - Checks component library supports selected frontend framework
   - Validates required CSS frameworks are configured
   - Examples:
     - shadcn-ui requires React + Tailwind CSS
     - Material-UI requires React
     - Ant Design supports React or Vue
     - Tremor requires React + Tailwind CSS

3. **State Management Compatibility** (`validate_state_management_compatibility`)
   - React: Zustand, Redux, MobX, Jotai, Valtio, Context API
   - Vue: Pinia, Vuex, Harlem
   - Prevents cross-framework state managers (e.g., Pinia with React)

4. **Meta-Framework Conflicts** (`validate_meta_framework_compatibility`)
   - Prevents selecting multiple conflicting meta-frameworks
   - Examples:
     - Next.js conflicts with: Vite, Remix, Gatsby, Astro, TanStack
     - Nuxt conflicts with: Vite, Quasar, VitePress
     - SvelteKit conflicts with: Vite, Elder.js

#### Example Errors

```bash
# Missing frontend framework
Component library selected but no frontend framework chosen. Select a frontend framework first.

# Component library incompatibility
shadcn-ui requires react framework, but vue was selected. shadcn/ui is built specifically for React with Tailwind CSS

# CSS framework mismatch
tremor requires tailwind CSS framework, but styled-components was selected. Tremor is built for React with Tailwind CSS

# State management incompatibility
pinia is not compatible with React. It's designed for vue.

# Meta-framework conflict
nextjs and remix are mutually exclusive. Choose one meta-framework.
```

#### Component Library Requirements

| Library | Framework | CSS Framework | Notes |
|---------|-----------|---------------|-------|
| shadcn-ui | React | Tailwind | React + Tailwind only |
| Material-UI | React | - | React-specific |
| Ant Design | React, Vue | - | Multi-framework |
| Chakra UI | React | - | React-specific |
| Mantine | React | - | React-specific |
| NextUI | React | - | React-specific |
| Headless UI | React, Vue | - | Multi-framework |
| Tremor | React | Tailwind | React + Tailwind only |
| Preline | - | Tailwind | Tailwind components |

### Build Tool Validator

**Location**: `template/hooks/validators/build_tool_validator.py`

**Purpose**: Validates build tool compatibility with frontend frameworks, backend languages, and project structure.

#### Validation Rules

1. **Framework Compatibility** (`_validate_framework_compatibility`)
   - Checks build tool supports selected frontend framework
   - Provides alternative suggestions if incompatible
   - Example: Turbopack only supports React (Next.js)

2. **React Framework Compatibility** (`_validate_react_framework_compatibility`)
   - Validates React meta-framework works with build tool
   - Next.js works with: Turbopack, Webpack
   - Remix works with: Vite, esbuild
   - Gatsby works with: Webpack

3. **Node.js Requirement** (`_validate_node_requirement`)
   - Warns if build tool requires Node.js but not selected
   - All build tools require Node.js (Vite, Webpack, Rollup, etc.)

4. **TypeScript Support** (`_check_typescript_support`)
   - Warns if build tool has limited TypeScript support
   - esbuild has limited plugin ecosystem for some frameworks

5. **Recommendations** (`_provide_recommendations`)
   - Monorepo projects: Vite, Rspack, Turbopack (better caching)
   - Performance-focused: esbuild, Rspack, Turbopack
   - Library builds: Rollup, esbuild, Vite
   - Simple apps: Parcel, Vite

#### Example Errors

```bash
# Framework incompatibility
Build tool 'turbopack' is not compatible with 'vue' framework. Compatible frameworks: react
Consider using one of these build tools instead: vite, webpack, rollup

# React framework warning
Build tool 'esbuild' may have limited support for 'gatsby'. Recommended tools: webpack

# Node.js requirement
Build tool 'vite' requires Node.js. Consider adding 'node' to backend_languages or choosing a different build tool.

# TypeScript support
Build tool 'esbuild' may have limited TypeScript support. Consider using Vite, Webpack, or Rspack for better TypeScript integration.

# Plugin ecosystem warning
esbuild has limited plugin support for vue. Consider using Vite or Rollup instead.
```

#### Build Tool Compatibility Matrix

| Build Tool | Frameworks | TypeScript | Node Required | Best For |
|------------|------------|------------|---------------|----------|
| Vite | React, Vue, Svelte, Solid | ✅ | ✅ | Modern apps, fast HMR |
| Webpack | React, Vue, Svelte, Solid, Angular | ✅ | ✅ | Mature ecosystem |
| esbuild | React, Solid | ⚠️ Limited | ✅ | Speed, simple builds |
| Rollup | React, Vue, Svelte, Solid | ✅ | ✅ | Library bundling |
| Rspack | React, Vue, Svelte, Solid | ✅ | ✅ | Webpack-compatible, fast |
| Turbopack | React (Next.js) | ✅ | ✅ | Next.js only |
| Farm | React, Vue, Svelte | ✅ | ✅ | Rust-based bundler |
| Parcel | React, Vue, Svelte, Solid | ✅ | ✅ | Zero-config |

### Monorepo Tool Validator

**Location**: `template/hooks/validators/monorepo_tool_validator.py`

**Purpose**: Validates monorepo tool compatibility with repository structure, package managers, and project scale.

#### Validation Rules

1. **Package Manager Compatibility** (`_validate_package_manager_compatibility`)
   - Turborepo supports: pnpm, npm, yarn, bun
   - Nx supports: pnpm, npm, yarn, bun
   - Lerna supports: pnpm, npm, yarn (no bun yet)
   - Moon is language-agnostic (all package managers)
   - Bazel is language-agnostic (doesn't use npm)

2. **Repository Structure** (`_validate_repository_structure`)
   - Warns if using flat/src-layout with monorepo tool
   - Recommends apps-packages or multi-workspace structures
   - Single-package structures don't benefit from monorepo tools

3. **Language Support** (`_validate_language_support`)
   - Turborepo: Node.js only
   - Nx: Node.js, Python, Rust, Go
   - Lerna: Node.js only
   - Moon: Node.js, Python, Rust, Go (multi-language focus)
   - Bazel: Node.js, Python, Rust, Go, Java (enterprise scale)

4. **Project Scale** (`_check_project_scale`)
   - Turborepo: 2-50 packages (good balance)
   - Nx: 5+ packages (feature-rich, best for larger projects)
   - Lerna: 3+ packages (publishing-focused)
   - Moon: 3+ packages (multi-language)
   - Bazel: 20+ packages (enterprise/Google scale)

5. **Feature Recommendations** (`_provide_feature_recommendations`)
   - Remote caching: Turborepo, Nx, Bazel
   - Affected commands: Nx, Moon, Bazel
   - Publishing: Lerna (conventional commits, changelogs)
   - Multi-language: Moon, Bazel, Nx

#### Example Errors

```bash
# Package manager incompatibility
Monorepo tool 'lerna' is not compatible with package manager 'bun'. Compatible managers: pnpm, npm, yarn
Consider using one of these monorepo tools instead: turborepo, nx, moon

# Structure mismatch
Repository structure 'flat' is not ideal for monorepo tool 'turborepo'. Consider using 'apps-packages' or 'multi-workspace' structure.

# Language support warning
Monorepo tool 'turborepo' is optimized for Node.js but your project uses: rust, python. Consider using 'moon' or 'bazel' for better multi-language support.

# Scale mismatch
Bazel is designed for very large monorepos (100+ packages). For smaller projects, consider 'turborepo' or 'nx' instead.

# No tool warning
Repository structure 'apps-packages' typically benefits from a monorepo tool. Consider using 'turborepo' or 'nx' for better task orchestration and caching.
```

#### Monorepo Tool Comparison

| Tool | Package Managers | Languages | Min Packages | Features |
|------|------------------|-----------|--------------|----------|
| Turborepo | pnpm, npm, yarn, bun | Node.js | 2 | Caching, pipelines, remote cache |
| Nx | pnpm, npm, yarn, bun | Node, Python, Rust, Go | 5 | Caching, affected, graph, generators, plugins |
| Lerna | pnpm, npm, yarn | Node.js | 3 | Versioning, publishing, conventional commits |
| Moon | All | Node, Python, Rust, Go | 3 | Caching, affected, multi-language, toolchain |
| Bazel | N/A | Node, Python, Rust, Go, Java | 20+ | Caching, hermetic, remote execution, multi-language |

### Tool Validator

**Location**: `template/hooks/validators/tool_validator.py`

**Purpose**: Validates development tool, package manager, build tool, monorepo tool, and IDE availability.

#### Validation Rules

1. **Language Tools** (`validate_language_tools`)
   - **Errors** if required language tools missing from PATH
   - Python: `python3`, `pip`
   - Rust: `cargo`, `rustc`
   - Node: `node`, `npm`
   - Go: `go`
   - Java: `java`, `javac`
   - C#: `dotnet`

2. **Package Managers** (`validate_package_managers`)
   - **Errors** if selected package manager not in PATH
   - Provides installation commands (e.g., `pip install poetry`)

3. **Build Tools** (`validate_build_tools`)
   - **Warnings only** (installed as project dependency)
   - Checks: vite, webpack, parcel, esbuild, rollup, turbopack, rspack, farm

4. **Monorepo Tools** (`validate_monorepo_tools`)
   - **Warnings only** (global install recommended)
   - Checks: turborepo, nx, rush, lerna, moon, bazel
   - Provides global install commands (e.g., `npm install -g turborepo`)

5. **IDE Tools** (`validate_ide_tools`)
   - **Informational only** (configs still generated)
   - Checks: vscode, cursor, zed, neovim, emacs, sublime, nova

#### Example Messages

```bash
# Language tool error
Required tool 'cargo' for rust not found in PATH. Install it before continuing.

# Package manager error
Python package manager 'poetry' (poetry) not found. Install it with: pip install poetry

# Build tool warning
Build tool 'vite' (vite) not found. It will be installed as a project dependency.

# Monorepo tool warning
Monorepo tool 'turborepo' (turbo) not found. Install it globally: npm install -g turborepo

# IDE informational
IDE 'cursor' command 'cursor' not found in PATH. Configuration files will still be generated.
```

#### Severity Levels

| Validation | Severity | Blocks Render | Purpose |
|------------|----------|---------------|---------|
| Language Tools | **Error** | ✅ Yes | Critical for compilation |
| Package Managers | **Error** | ✅ Yes | Critical for dependencies |
| Build Tools | **Warning** | ❌ No | Installed as dependency |
| Monorepo Tools | **Warning** | ❌ No | Global install recommended |
| IDE Tools | **Info** | ❌ No | Optional convenience |

### Structure Validator

**Location**: `template/hooks/validators/structure_validator.py`

**Purpose**: Validates repository structure compatibility with project layout, languages, tools, and service count.

#### Validation Rules

1. **Structure Compatibility** (`validate_structure_compatibility`)
   - **Errors** if structure requires monorepo but single-package selected
   - apps-packages and multi-workspace require `project_layout=monorepo`
   - **Errors** if flat/src-layout used with monorepo layout

2. **Language Compatibility** (`validate_structure_language_compatibility`)
   - **Warnings** if multiple languages with single-language structure
   - src-layout is Python-optimized (PEP 517/518)
   - Multi-language projects benefit from apps-packages or multi-workspace

3. **Tool Configuration** (`validate_structure_tools`)
   - **Warnings** if monorepo structure without monorepo tool
   - **Warnings** if multi-workspace without workspace-enabled package manager (pnpm/yarn)

4. **Service Count** (`validate_structure_service_count`)
   - Estimates service count from configuration (CLI, API, frontend, docs)
   - **Warnings** if structure mismatch with scale:
     - flat/src-layout: max 1 service
     - apps-packages/multi-workspace: min 2 services
     - domain-driven: min 2 domains

#### Example Errors

```bash
# Structure requires monorepo
Repository structure 'apps-packages' requires project_layout='monorepo', but 'single-package' was selected. Change project_layout to 'monorepo'.

# Incompatible with monorepo
Repository structure 'src-layout' is not recommended for monorepos. Consider using 'apps-packages' or 'multi-workspace' instead.

# Multiple languages warning
Multiple backend languages (python, rust, node) selected with 'src-layout' structure. Consider 'apps-packages' or 'multi-workspace' for better polyglot project organization.

# Missing monorepo tool
Repository structure 'apps-packages' works best with a monorepo tool. Consider enabling: turborepo, nx

# Package manager mismatch
'multi-workspace' works best with: pnpm, yarn. You selected 'npm' which may have limited workspace support.

# Scale mismatch
Repository structure 'apps-packages' is designed for at least 2 services, but only 1 detected. Consider 'src-layout' for simpler projects.

# Over-engineered
Repository structure 'flat' is not ideal for 4 services. Consider 'apps-packages' or 'multi-workspace' for better organization.
```

#### Structure Requirements

| Structure | Best For | Requires | Min Services | Recommended Tools |
|-----------|----------|----------|--------------|-------------------|
| flat | Small, single-purpose | - | Max 1 | - |
| src-layout | Python packages (PEP 517/518) | - | 1 | Python-focused |
| apps-packages | Monorepos with services + packages | Monorepo layout | 2+ | Turborepo, Nx |
| multi-workspace | Large polyglot monorepos | Monorepo layout | 2+ | pnpm, yarn workspaces |
| domain-driven | Enterprise, domain-organized | - | 2+ domains | Nx, Bazel |

## Validator Composition and Precedence

### Execution Order

Validators run in this sequence within the pre-generation hook:

1. **Tool Provisioning** (pre_gen_project.py)
   - Auto-installs uv, Node.js, pnpm via mise
   - Logs all attempts to `.riso/toolchain_provisioning.jsonl`

2. **SaaS Starter Validation** (pre_gen_project.py)
   - Only if `saas_starter_module=enabled`
   - Validates database/storage/hosting combinations
   - Severity levels: error (blocks), warning (continues), info (notes)

3. **Domain Validators** (called from quality_tool_check.py or custom scripts)
   - **Language Validator**: Backend languages, frameworks, package managers
   - **Frontend Validator**: Frontend frameworks, components, state management
   - **Build Tool Validator**: Build tool compatibility
   - **Monorepo Tool Validator**: Monorepo tool compatibility
   - **Tool Validator**: Tool availability checks
   - **Structure Validator**: Repository structure validation

4. **Quality Tool Provisioning** (pre_gen_project.py)
   - Python tools: ruff, ty, pylint, pytest
   - Node tools: eslint, prettier, typescript (if needed)

### Error Aggregation

Each validator returns structured results:

```python
# Simple validators (language, frontend)
def validate(answers: Dict[str, Any]) -> List[str]:
    """Returns list of error messages"""
    errors = []
    errors.extend(validate_check1(answers))
    errors.extend(validate_check2(answers))
    return errors

# Advanced validators (build_tool, monorepo_tool, structure, tool)
def validate(answers: Dict[str, Any]) -> Dict[str, List[str]]:
    """Returns dict with severity levels"""
    return {
        "errors": [...],      # Block render
        "warnings": [...],    # Continue with warnings
        "info": [...]         # Informational only
    }
```

### Precedence Rules

1. **Errors always block** - Any validator returning errors halts rendering
2. **Warnings are logged** - Displayed but don't block
3. **Info is informational** - Logged for user awareness
4. **First failure wins** - Validators stop at first error in sequence
5. **Auto-healing attempted first** - mise provisioning before manual errors

### Cross-Validator Dependencies

Some validators depend on others' results:

```
Language Validator
    ↓ (provides backend_languages)
Frontend Validator
    ↓ (provides frontend_framework)
Build Tool Validator
    ↓ (provides build_tool)
Monorepo Tool Validator
    ↓ (provides monorepo_tool)
Structure Validator (uses all above)
```

Example dependency chain:
1. Language validator ensures Python is available
2. Frontend validator checks React is selected
3. Build Tool validator recommends Vite for React
4. Monorepo Tool validator checks if Turborepo compatible with pnpm
5. Structure validator ensures apps-packages structure with 2+ services

## Error Resolution Guide

### Common Error Scenarios

#### Scenario 1: Missing Language Tool

**Error:**
```bash
Python requires 'python3' but it's not found in PATH. Install it before continuing.
```

**Resolution:**
```bash
# macOS (Homebrew)
brew install python@3.11

# Ubuntu/Debian
sudo apt install python3.11 python3-pip

# Windows (winget)
winget install Python.Python.3.11

# Verify
python3 --version  # Should show 3.11+
```

#### Scenario 2: Package Manager Missing

**Error:**
```bash
Python package manager 'uv' (uv) not found in PATH. Install it or select a different package manager.
```

**Resolution:**
```bash
# Option 1: Install uv
curl -LsSf https://astral.sh/uv/install.sh | sh

# Option 2: Change to poetry
# Update copier-answers.yml:
python_package_manager: poetry

# Install poetry
pip install poetry

# Option 3: Auto-provision with mise
mise install uv@0.4
```

#### Scenario 3: Framework Incompatibility

**Error:**
```bash
shadcn-ui requires react framework, but vue was selected. shadcn/ui is built specifically for React with Tailwind CSS
```

**Resolution:**
```bash
# Option 1: Change component library
component_library: ant-design  # Supports both React and Vue

# Option 2: Change to React
frontend_framework: react
component_library: shadcn-ui
css_framework: tailwind

# Re-run copier
copier copy . output/ --force
```

#### Scenario 4: Meta-Framework Conflict

**Error:**
```bash
nextjs and remix are mutually exclusive. Choose one meta-framework.
```

**Resolution:**
```yaml
# Option 1: Keep Next.js
frontend_framework: react
react_framework: nextjs
build_tool: turbopack  # or webpack

# Option 2: Switch to Remix
frontend_framework: react
react_framework: remix
build_tool: vite  # or esbuild

# Don't select both nextjs AND remix
```

#### Scenario 5: Monorepo Tool Incompatibility

**Error:**
```bash
Monorepo tool 'lerna' is not compatible with package manager 'bun'. Compatible managers: pnpm, npm, yarn
```

**Resolution:**
```yaml
# Option 1: Change package manager
node_package_manager: pnpm
monorepo_tool: lerna

# Option 2: Change monorepo tool
node_package_manager: bun
monorepo_tool: turborepo  # Supports bun

# Option 3: Use language-agnostic tool
monorepo_tool: moon  # Supports all package managers
```

#### Scenario 6: Structure Mismatch

**Error:**
```bash
Repository structure 'apps-packages' requires project_layout='monorepo', but 'single-package' was selected. Change project_layout to 'monorepo'.
```

**Resolution:**
```yaml
# Option 1: Use monorepo layout
project_layout: monorepo
repository_structure: apps-packages
monorepo_tool: turborepo

# Option 2: Simplify structure
project_layout: single-package
repository_structure: src-layout  # Python-optimized
# or
repository_structure: flat  # Simple projects
```

#### Scenario 7: SaaS Starter Incompatibility

**Error:**
```bash
Cannot use Neon database with Supabase Storage. Choose either:
  1. Full Supabase (database + storage)
  2. Neon database + Cloudflare R2 storage
```

**Resolution:**
```yaml
# Option 1: Full Supabase
saas_database: supabase
saas_storage: supabase-storage

# Option 2: Neon + R2
saas_database: neon
saas_storage: r2

# Option 3: Supabase DB + R2 (also valid)
saas_database: supabase
saas_storage: r2
```

### Debugging Validation Failures

#### 1. Review Provisioning Log

```bash
# Check what tools were attempted
cat .riso/toolchain_provisioning.jsonl | jq '.'

# Filter by status
cat .riso/toolchain_provisioning.jsonl | jq 'select(.status == "failed")'

# See retry commands
cat .riso/toolchain_provisioning.jsonl | jq -r '.retry_command // empty'
```

#### 2. Manually Verify Configuration

```bash
# Check Copier context
echo $COPIER_ANSWERS | jq '.'

# Validate specific answer
echo $COPIER_ANSWERS | jq '.backend_languages'

# Check if tools available
which python3 uv node pnpm cargo
```

#### 3. Run Validators Standalone

```python
# scripts/debug_validators.py
import json
import sys
from pathlib import Path

sys.path.insert(0, str(Path(__file__).parent.parent / "template" / "hooks"))

from validators.language_validator import validate as validate_language
from validators.frontend_validator import validate as validate_frontend

# Load answers
with open("copier-answers.yml") as f:
    import yaml
    answers = yaml.safe_load(f)

# Test individual validators
lang_errors = validate_language(answers)
print(f"Language errors: {lang_errors}")

frontend_errors = validate_frontend(answers)
print(f"Frontend errors: {frontend_errors}")
```

#### 4. Enable Verbose Logging

```bash
# Set debug mode
export COPIER_DEBUG=1

# Run with more context
copier copy --vcs-ref HEAD . output/ -v

# Check stderr for validator output
copier copy . output/ 2>&1 | tee validation.log
```

- React + Vite recommended combination

### Example 2: Rust Axum + Vue + Nuxt

```yaml
# Rust backend with Vue frontend
project_name: "Axum Nuxt App"
project_layout: monorepo
repository_structure: apps-packages

backend_languages: [rust]
rust_web_framework: axum
rust_async_runtime: tokio

frontend_framework: vue
```

**Error:**

```bash
shadcn-ui requires react framework, but vue was selected.
```

```bash
tremor requires tailwind CSS framework, but styled-components was selected.
```

## Extending Validators

### Adding a New Validator

**Step 1: Create Validator Module**

```python
# template/hooks/validators/database_validator.py
"""Database validator for database compatibility checks."""

from typing import Any, Dict, List

# Database requirements
DATABASE_REQUIREMENTS = {
    "postgresql": {
        "requires_driver": True,
        "python_drivers": ["psycopg2", "asyncpg"],
        "node_drivers": ["pg", "postgres"],
    },
    "mysql": {
        "requires_driver": True,
        "python_drivers": ["pymysql", "aiomysql"],
        "node_drivers": ["mysql2"],
    },
    "sqlite": {
        "requires_driver": False,
        "python_drivers": ["sqlite3"],  # Built-in
        "node_drivers": ["better-sqlite3"],
    },
}

def validate_database_drivers(answers: Dict[str, Any]) -> List[str]:
    """Validate database drivers are configured."""
    errors = []
    database = answers.get("database")
    backend_languages = answers.get("backend_languages", [])

    if not database or database == "none":
        return errors

    db_config = DATABASE_REQUIREMENTS.get(database, {})

    # Check Python driver
    if "python" in backend_languages:
        python_orm = answers.get("python_orm")
        if not python_orm and db_config.get("requires_driver"):
            errors.append(
                f"Database '{database}' requires an ORM/driver. "
                f"Select one of: {', '.join(db_config['python_drivers'])}"
            )

    return errors

def validate(answers: Dict[str, Any]) -> List[str]:
    """Run all database validations."""
    errors = []
    errors.extend(validate_database_drivers(answers))
    return errors
```

### Creating Severity-Aware Validators

**Advanced Validator with Error/Warning/Info:**

```python
# template/hooks/validators/performance_validator.py
"""Performance validator for performance-impacting configurations."""

from typing import Any, Dict, List

def validate_performance_choices(answers: Dict[str, Any]) -> Dict[str, List[str]]:
    """Validate performance-related choices."""
    result = {
        "errors": [],
        "warnings": [],
        "info": []
    }

    build_tool = answers.get("build_tool")
    bundle_size_limit = answers.get("bundle_size_limit")

    # Error: No build tool for production
    if answers.get("environment") == "production" and build_tool == "none":
        result["errors"].append(
            "Production builds require a build tool for optimization. "
            "Select vite, webpack, or rspack."
        )

    # Warning: Slow build tool for large projects
    if build_tool == "webpack" and answers.get("project_scale") == "large":
        result["warnings"].append(
            "Webpack may be slow for large projects. "
            "Consider rspack (webpack-compatible, faster) or vite."
        )

    # Info: Performance optimization available
    if bundle_size_limit and bundle_size_limit > 500:
        result["info"].append(
            "Bundle size limit >500KB. Consider code splitting, "
            "tree shaking, and lazy loading for better performance."
        )

    return result

def validate(answers: Dict[str, Any]) -> Dict[str, List[str]]:
    """Run all performance validations."""
    return validate_performance_choices(answers)
```

## Best Practices

### For Validator Authors

1. **Fail Fast with Clear Messages**
   ```python
   # Good: Specific, actionable
   errors.append(
       "shadcn-ui requires React + Tailwind CSS, but Vue + styled-components "
       "was selected. Change to: frontend_framework=react, css_framework=tailwind"
   )

   # Bad: Vague, no guidance
   errors.append("Invalid component library configuration")
   ```

2. **Use Severity Appropriately**
   - **Errors**: Breaks functionality, must fix (missing tools, incompatible choices)
   - **Warnings**: Works but suboptimal (wrong scale, performance issues)
   - **Info**: Optional improvements (available features, tips)

3. **Provide Alternatives**
   ```python
   if build_tool not in compatible_tools:
       errors.append(f"Build tool '{build_tool}' incompatible with '{framework}'")
       # Suggest alternatives
       alternatives = [tool for tool, frameworks in COMPATIBILITY.items()
                      if framework in frameworks]
       if alternatives:
           errors.append(f"Try: {', '.join(alternatives)}")
   ```

4. **Test Edge Cases**
   - Empty/None values
   - Multiple selections (arrays)
   - Case sensitivity
   - Default values

5. **Document Requirements**
   ```python
   # Add module docstring
   """Database validator for database compatibility.

   Validates:
   - Database driver availability
   - ORM compatibility with database
   - Connection pool configuration
   - Migration tool compatibility

   Example errors:
   - PostgreSQL requires driver: psycopg2 or asyncpg
   - SQLAlchemy incompatible with MongoDB (use Motor or PyMongo)
   """
   ```

### For Template Users

1. **Read Error Messages Completely**
   - Errors include fix instructions
   - Check suggested alternatives
   - Look for installation commands

2. **Check Provisioning Log**
   ```bash
   cat .riso/toolchain_provisioning.jsonl | jq '.[] | select(.status == "failed")'
   ```

3. **Verify Tool Availability**
   ```bash
   # Before running copier
   which python3 node pnpm uv cargo

   # Check versions
   python3 --version  # 3.11+
   node --version     # 20+
   pnpm --version     # 8+
   ```

4. **Use Compatible Combinations**
   - Start with documented examples
   - Add complexity incrementally
   - Test locally before CI

5. **Enable Auto-Provisioning**
   ```bash
   # Install mise for auto-provisioning
   curl https://mise.run | sh

   # Configure mise
   mise install uv@0.4 node@20 pnpm@8

   # Copier will auto-provision via mise
   copier copy . output/
   ```

## Summary

The Riso validator system ensures:

- **Tool Availability**: Required language tools, package managers, and build tools are present
- **Configuration Compatibility**: Framework, library, and tool selections work together
- **Project Structure**: Repository structure matches project scale and complexity
- **Auto-Healing**: Single-pass provisioning attempts before manual errors
- **Actionable Guidance**: Every error includes fix instructions or alternatives

**Key Takeaways:**

1. Validators run during pre-generation, blocking invalid configurations
2. Six specialized validators cover languages, frontend, build tools, monorepo, tools, and structure
3. Severity levels (error/warning/info) control blocking behavior
4. Errors include installation commands, alternative suggestions, and fix guidance
5. Extensible system allows custom validators for new modules
6. Provisioning log (`.riso/toolchain_provisioning.jsonl`) tracks all attempts

For implementation details, see individual validator source files in `template/hooks/validators/`.
