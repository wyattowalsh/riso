## Overview

Riso supports 14 SaaS categories with 37+ provider options, enabling you to build production-ready applications with best-in-class services. This guide helps you navigate the complex interdependencies and choose the optimal stack for your use case.

### Technology Categories

| Category | Options | Purpose |
|----------|---------|---------|
| **Runtime** | Next.js 16, Remix 2 | Application framework |
| **Hosting** | Vercel, Cloudflare | Deployment platform |
| **Database** | Neon, Supabase | Postgres database |
| **ORM** | Prisma, Drizzle | Database toolkit |
| **Auth** | Clerk, AuthJS | User authentication |
| **Billing** | Stripe, Paddle | Payment processing |
| **Jobs** | Trigger.dev, Inngest | Background tasks |
| **Email** | Resend, Postmark | Transactional email |
| **Analytics** | PostHog, Amplitude | Product analytics |
| **AI** | OpenAI, Anthropic | AI/LLM integration |
| **Storage** | R2, Supabase Storage | File storage |
| **CI/CD** | GitHub Actions, Cloudflare CI | Deployment automation |

## Compatibility Matrix

### Critical Incompatibilities

These combinations **cannot** be used together:

| Provider A | Provider B | Severity | Reason | Solution |
|------------|------------|----------|--------|----------|
| Neon | Supabase Storage | ERROR | Storage requires Supabase DB | Use Supabase (DB + Storage) OR Neon + R2 |

### Compatibility Warnings

These combinations work but have trade-offs:

| Provider A | Provider B | Impact | Workaround | Recommendation |
|------------|------------|--------|------------|----------------|
| Cloudflare | Prisma | Requires Data Proxy | Add `@prisma/client/edge` + Prisma Data Proxy | Use Drizzle ORM instead |
| Vercel | R2 | Egress charges | Use Cloudflare CDN | Accept egress costs or use Vercel Blob |

### Information Notices

| Provider A | Provider B | Note |
|------------|------------|------|
| Supabase | Clerk | Supabase Auth disabled in favor of Clerk |
| Clerk | WorkOS | WorkOS handles enterprise SSO, Clerk handles consumer auth |

### Required Dependencies

| Feature | Requires | Reason |
|---------|----------|--------|
| Billing (Stripe/Paddle) | Auth (Clerk/AuthJS) | Customer identity needed |
| Database | ORM (recommended) | Better developer experience |
| Background Jobs | Database | Job persistence |

## Migration Paths

### Database Migration

#### Supabase → Neon

**Reason**: Better edge performance, lower cost at scale.

```bash
# 1. Export Supabase database
pg_dump $SUPABASE_URL > supabase_dump.sql

# 2. Create Neon database
neon projects create --name my-app

# 3. Import to Neon
psql $NEON_URL < supabase_dump.sql

# 4. Migrate storage (Supabase Storage → R2)
node scripts/migrate-storage-to-r2.js

# 5. Update environment variables
# DATABASE_URL=$NEON_URL
# Remove SUPABASE_* vars
```

**Downtime**: ~15 minutes (for read-only mode + cutover)

**Gotchas**:
- Supabase Auth → Clerk migration requires user data export
- Realtime subscriptions need alternative (Inngest, websockets)
- Storage migration is one-way (R2 doesn't support Supabase transforms)

#### Neon → Supabase

**Reason**: Need realtime features, storage integration.

```bash
# 1. Create Supabase project
supabase projects create my-app

# 2. Export Neon database
pg_dump $NEON_URL > neon_dump.sql

# 3. Import to Supabase
psql $SUPABASE_URL < neon_dump.sql

# 4. Enable Supabase features
supabase db push
supabase storage create-bucket uploads
```

### ORM Migration

#### Prisma → Drizzle

**Reason**: Better edge compatibility, smaller bundle size.

**Step 1: Install Drizzle**
```bash
pnpm add drizzle-orm @neondatabase/serverless
pnpm add -D drizzle-kit
```

**Step 2: Convert Schema**
```typescript
// Before (Prisma)
model User {
  id        String   @id @default(cuid())
  email     String   @unique
  name      String?
  createdAt DateTime @default(now())
}

// After (Drizzle)
import { pgTable, text, timestamp } from 'drizzle-orm/pg-core';

export const users = pgTable('users', {
  id: text('id').primaryKey().$defaultFn(() => createId()),
  email: text('email').notNull().unique(),
  name: text('name'),
  createdAt: timestamp('created_at').notNull().defaultNow(),
});
```

**Step 3: Migrate Queries**
```typescript
// Before (Prisma)
const user = await prisma.user.findUnique({
  where: { email: 'user@example.com' },
});

// After (Drizzle)
const user = await db.query.users.findFirst({
  where: eq(users.email, 'user@example.com'),
});
```

**Migration Time**: 2-8 hours (depending on schema complexity)

**Automation Script**:
```bash
# Convert Prisma schema to Drizzle
npx prisma-to-drizzle schema.prisma > drizzle.schema.ts
```

#### Drizzle → Prisma

**Reason**: Need Prisma Client features (middleware, metrics).

Similar process in reverse, use `drizzle-to-prisma` converter.

### Auth Migration

#### AuthJS → Clerk

**Reason**: Better UX, pre-built components, org management.

**Data Migration**:
```typescript
// 1. Export users from AuthJS
const users = await prisma.user.findMany({
  include: { accounts: true },
});

// 2. Import to Clerk
for (const user of users) {
  await clerkClient.users.createUser({
    emailAddress: [user.email],
    firstName: user.name?.split(' ')[0],
    lastName: user.name?.split(' ')[1],
    publicMetadata: { legacyId: user.id },
  });
}
```

**Downtime**: ~1 hour (user re-authentication required)

### Storage Migration

#### R2 → Supabase Storage

**Reason**: Need image transformations, simpler auth model.

```typescript
import { r2, supabase } from '@/lib/clients';

async function migrateFiles() {
  const { files } = await r2.listFiles({ prefix: 'uploads/' });

  for (const file of files) {
    // Download from R2
    const buffer = await r2.downloadFile(file.key);

    // Upload to Supabase
    await supabase.uploadFile({
      path: file.key,
      file: buffer,
      contentType: file.contentType,
    });

    console.log(`Migrated: ${file.key}`);
  }
}
```

**Migration Time**: Depends on file count (10K files = ~1 hour)

## Cost Optimization Strategies

### 1. Database Costs

#### Neon Autoscaling

```yaml
# neon.toml
[compute]
autoscaling_limit_min_cu = 0.25  # Sleep to zero
autoscaling_limit_max_cu = 2     # Cap at 2 CU
autosuspend_delay_seconds = 300  # 5 min idle timeout
```

**Savings**: ~70% reduction vs always-on instance

#### Connection Pooling

```typescript
// Use Neon serverless driver (HTTP-based)
import { neon } from '@neondatabase/serverless';

const sql = neon(process.env.DATABASE_URL!);

// Or use connection pooling
const pooledUrl = process.env.DATABASE_URL!.replace(
  'postgres://',
  'postgres-pooler://'
);
```

**Impact**: Reduce connection overhead by 90%

#### Query Optimization

```typescript
// ❌ N+1 query problem
const users = await db.query.users.findMany();
for (const user of users) {
  const posts = await db.query.posts.findMany({
    where: eq(posts.userId, user.id),
  });
}

// ✅ Use joins
const users = await db.query.users.findMany({
  with: { posts: true },
});
```

**Impact**: 100x faster, 10x cheaper

### 2. Storage Costs

#### R2 vs S3 Cost Comparison

| Operation | R2 | S3 Standard | Savings |
|-----------|-----|-------------|---------|
| Storage | $0.015/GB | $0.023/GB | 35% |
| Egress | $0 | $0.09/GB | 100% |
| Class A Ops | $4.50/M | $5.00/M | 10% |

**Example**: 100GB storage + 1TB egress/month
- R2: $1.50 storage + $0 egress = **$1.50**
- S3: $2.30 storage + $90 egress = **$92.30**
- **Savings: $90.80/month (98%)**

#### Image Optimization

```typescript
// Use Cloudflare Image Transformations with R2
const optimizedUrl = `https://imagedelivery.net/${ACCOUNT_HASH}/${IMAGE_ID}/public`;

// Or use Supabase image transforms
const url = supabase.getPublicUrlWithTransform('avatar.jpg', {
  width: 300,
  height: 300,
  quality: 80,
  format: 'webp',
});
```

**Impact**: 60-80% bandwidth reduction

### 3. AI/LLM Costs

#### Token Usage Optimization

```typescript
// ❌ Sending full conversation history
const completion = await openai.chat.completions.create({
  model: 'gpt-4',
  messages: allMessages, // 10K tokens
});

// ✅ Summarize old messages
const completion = await openai.chat.completions.create({
  model: 'gpt-4',
  messages: [
    { role: 'system', content: summaryOfOldMessages },
    ...recentMessages,
  ], // 2K tokens
});
```

**Savings**: 80% token reduction

#### Model Selection

| Model | Cost/1M Tokens | Use Case |
|-------|----------------|----------|
| GPT-4o | $2.50 in / $10 out | Complex reasoning |
| GPT-4o-mini | $0.15 in / $0.60 out | Simple tasks |
| Claude 3.5 Sonnet | $3 in / $15 out | Long context |
| Claude 3.5 Haiku | $0.25 in / $1.25 out | Fast, simple tasks |

**Strategy**: Use cheap models for classification, expensive for generation.

#### Prompt Caching

```typescript
// Anthropic prompt caching (50% cost reduction)
const response = await anthropic.messages.create({
  model: 'claude-3-5-sonnet-20241022',
  system: [
    {
      type: 'text',
      text: LONG_SYSTEM_PROMPT, // Cached
      cache_control: { type: 'ephemeral' },
    },
  ],
  messages: [{ role: 'user', content: userQuery }],
});
```

**Savings**: 90% cost reduction for repeated system prompts

### 4. Email Costs

#### Batch Sending

```typescript
// ❌ Individual sends
for (const user of users) {
  await resend.emails.send({ to: user.email, subject, html });
}

// ✅ Batch API (50% cost reduction)
await resend.batch.send(
  users.map(user => ({ to: user.email, subject, html }))
);
```

#### Transactional vs Marketing

| Provider | Transactional | Marketing | Savings |
|----------|---------------|-----------|---------|
| Resend | $1/1K emails | N/A | - |
| SendGrid | $1.95/1K | $0.08/1K | 96% |
| Postmark | $1.25/1K | N/A | - |

**Strategy**: Use SendGrid Marketing Cloud for bulk emails.

### 5. Monitoring Costs

#### Sampling Strategy

```typescript
// Send 10% of traces to Sentry
Sentry.init({
  tracesSampleRate: 0.1,
  profilesSampleRate: 0.1,
});

// Use PostHog session recording sparingly
posthog.init(POSTHOG_KEY, {
  session_recording: {
    recordCrossOriginIframes: false,
    maskAllInputs: true,
    sampleRate: 0.1, // Record 10% of sessions
  },
});
```

**Impact**: 90% cost reduction, maintains statistical significance

## Troubleshooting

### Common Issues

#### 1. Prisma on Cloudflare Workers

**Error**: `PrismaClient is unable to run in Cloudflare Workers`

**Solution**: Use Drizzle ORM or Prisma Data Proxy

```typescript
// Option A: Switch to Drizzle (recommended)
import { drizzle } from 'drizzle-orm/neon-http';
const db = drizzle(process.env.DATABASE_URL);

// Option B: Use Prisma Data Proxy (adds cost + latency)
// DATABASE_URL=prisma://aws-us-east-1.prisma-data.com/?api_key=xxx
```

#### 2. Neon + Supabase Storage

**Error**: `Supabase Storage requires Supabase Database`

**Solution**: Choose one of:
- Full Supabase: `database: supabase` + `storage: supabase-storage`
- Neon + R2: `database: neon` + `storage: r2`

#### 3. Clerk + Supabase Auth Conflict

**Warning**: Both Clerk and Supabase provide authentication

**Solution**: Disable Supabase Auth in dashboard:
1. Go to Supabase Dashboard → Authentication
2. Disable "Enable email confirmations"
3. Use Clerk as single source of truth for auth

#### 4. High R2 Egress from Vercel

**Issue**: R2 egress is free within Cloudflare network, but costs apply from Vercel

**Solutions**:
1. Use Cloudflare CDN in front of R2
2. Switch to Vercel Blob (if budget allows)
3. Accept egress costs (still cheaper than S3)

#### 5. Background Jobs Not Running

**Debugging**:
```typescript
// Check Inngest dashboard
await inngest.send({
  name: 'test/job',
```
