GraphQL Production Deployment Guide¶

Comprehensive guide for deploying and migrating GraphQL APIs built with the Riso template’s Strawberry GraphQL module.

Overview¶

The Riso GraphQL module provides a production-ready GraphQL API using:

  • Strawberry GraphQL: Type-safe schema with Python dataclasses

  • FastAPI Integration: High-performance async server

  • JWT Authentication: Token-based security with configurable validation

  • DataLoader Pattern: N+1 query prevention with automatic batching

  • Query Protection: Complexity and depth limits to prevent abuse

  • Environment-based Config: Pydantic Settings for 12-factor compliance

Prerequisites¶

Before deploying to production:

  • Python 3.11+ with uv package manager

  • FastAPI with uvicorn server

  • PostgreSQL or compatible database (optional but recommended)

  • Redis for caching/sessions (recommended)

  • Secrets management solution (AWS Secrets Manager, Vault, etc.)

Production Deployment Checklist¶

1. Security Configuration¶

JWT Secret Key Management¶

CRITICAL: Never use the default secret key in production.

Generate a secure secret key:

# Generate a cryptographically secure 64-character secret
python3 -c "import secrets; print(secrets.token_urlsafe(64))"

# Or use openssl
openssl rand -base64 64 | tr -d '\n'

Set via environment variable:

export RISO_GRAPHQL_JWT_SECRET_KEY="your-generated-secret-key-here"
export RISO_GRAPHQL_JWT_ALGORITHM="HS256"
export RISO_GRAPHQL_JWT_EXPIRATION_MINUTES=30

Production best practices:

  • Store secrets in encrypted key vault (AWS Secrets Manager, HashiCorp Vault)

  • Rotate keys quarterly at minimum

  • Use separate keys per environment (dev/staging/production)

  • Implement key versioning for zero-downtime rotation

  • Never commit secrets to version control

Key rotation procedure:

# Example key rotation with grace period
# config.py - add to GraphQLAuthSettings

@property
def active_jwt_keys(self) -> list[str]:
    """
    Support multiple keys during rotation.
    First key used for signing, all keys attempted for verification.
    """
    keys = [self.jwt_secret_key]  # Current primary key

    # Add previous key for grace period (from environment)
    if previous_key := os.getenv("RISO_GRAPHQL_JWT_SECRET_KEY_PREVIOUS"):
        keys.append(previous_key)

    return keys

3. Security Hardening¶

Rate Limiting¶

Implement rate limiting middleware:

# middleware/rate_limit.py
from fastapi import Request, HTTPException
from slowapi import Limiter
from slowapi.util import get_remote_address

limiter = Limiter(key_func=get_remote_address)

# Apply to GraphQL endpoint
@app.post("/graphql")
@limiter.limit("100/minute")  # 100 requests per minute per IP
async def graphql_endpoint(request: Request):
    return await graphql_app(request)

Advanced rate limiting with Redis:

# middleware/redis_rate_limit.py
import time
from fastapi import Request, HTTPException
import redis.asyncio as redis

redis_client = redis.from_url("redis://localhost:6379")

async def rate_limit_middleware(request: Request, call_next):
    """
    Token bucket rate limiting with Redis.
    Allows burst traffic while maintaining average rate.
    """
    user_id = request.state.user_id or get_remote_address(request)
    key = f"rate_limit:{user_id}"

    # Token bucket algorithm
    max_tokens = 100
    refill_rate = 10  # tokens per second

    async with redis_client.pipeline() as pipe:
        now = time.time()

        # Get current bucket state
        bucket = await redis_client.hgetall(key)

        if bucket:
            tokens = float(bucket[b"tokens"])
            last_refill = float(bucket[b"last_refill"])

            # Refill tokens based on elapsed time
            elapsed = now - last_refill
            tokens = min(max_tokens, tokens + elapsed * refill_rate)
        else:
            tokens = max_tokens

        # Try to consume one token
        if tokens >= 1:
            tokens -= 1
            await redis_client.hset(key, mapping={
                "tokens": tokens,
                "last_refill": now,
            })
            await redis_client.expire(key, 3600)  # 1 hour TTL

            return await call_next(request)
        else:
            raise HTTPException(
                status_code=429,
                detail="Rate limit exceeded. Try again later.",
            )

Introspection Control¶

Disable in production to prevent schema discovery:

# Production setting
export RISO_GRAPHQL_ENABLE_INTROSPECTION=false

# Schema introspection queries will return:
# "GraphQL introspection is disabled in production"

Selective introspection (allow for admin users only):

# Custom introspection permission in schema.py
from strawberry.extensions import SchemaExtension

class ConditionalIntrospectionExtension(SchemaExtension):
    def on_operation(self):
        # Check if operation is introspection query
        if self.execution_context.operation_name == "IntrospectionQuery":
            user = self.execution_context.context.current_user

            # Allow only for admin users
            if not user or "admin" not in user.get("permissions", []):
                raise AuthError("Introspection is disabled")

        yield

# Add to schema
schema = strawberry.Schema(
    query=Query,
    mutation=Mutation,
    extensions=[ConditionalIntrospectionExtension],
)

4. Performance Optimization¶

DataLoader Configuration¶

Template includes DataLoader implementation. Optimize for your data access patterns:

# dataloaders.py - production configuration
from aiodataloader import DataLoader

class UserDataLoader(DataLoader):
    """
    Production optimizations:
    - Batch size limits (prevent unbounded IN queries)
    - Cache TTL (prevent stale data)
    - Error handling (partial failures)
    """

    def __init__(self, db: AsyncSession, cache_ttl: int = 60):
        super().__init__(
            batch_load_fn=self.batch_load_users,
            max_batch_size=100,  # Limit batch size
            cache=True,          # Enable per-request caching
        )
        self.db = db
        self.cache_ttl = cache_ttl

    async def batch_load_users(self, user_ids: List[str]) -> List[Optional[User]]:
        """
        Batch load with database query optimization.
        """
        # Use SELECT IN with parameter limit
        if len(user_ids) > 1000:
            # Split into multiple batches if needed
            logger.warning(f"Large batch size: {len(user_ids)} users")

        result = await self.db.execute(
            select(UserModel)
            .where(UserModel.id.in_(user_ids))
            .options(selectinload(UserModel.posts))  # Eager load relations
        )

        users_dict = {user.id: User.from_orm(user) for user in result.scalars()}

        # Return in same order as requested (important!)
        return [users_dict.get(uid) for uid in user_ids]

Database Query Optimization¶

Use SQLAlchemy async patterns:

# Efficient query patterns for GraphQL resolvers

# ❌ BAD: N+1 queries
async def get_users_with_posts(info: Info) -> List[User]:
    users = await db.execute(select(UserModel))
    for user in users:
        # This triggers N additional queries!
        posts = await db.execute(
            select(PostModel).where(PostModel.author_id == user.id)
        )
        user.posts = posts

# ✅ GOOD: Single query with eager loading
async def get_users_with_posts(info: Info) -> List[User]:
    result = await db.execute(
        select(UserModel).options(
            selectinload(UserModel.posts)  # Eager load in same query
        )
    )
    return result.scalars().all()

# ✅ BEST: Use DataLoader for dynamic loading
async def resolve_user_posts(user: User, info: Info) -> List[Post]:
    return await info.context.dataloaders.posts_by_author_loader.load(user.id)

Index database columns used in GraphQL filters:

-- Add indexes for common GraphQL queries
CREATE INDEX idx_users_email ON users(email);
CREATE INDEX idx_posts_author_id ON posts(author_id);
CREATE INDEX idx_posts_created_at ON posts(created_at DESC);

-- Composite indexes for common filter combinations
CREATE INDEX idx_posts_author_created ON posts(author_id, created_at DESC);

Response Caching¶

Implement field-level caching:

# Install strawberry cache extension
# uv add strawberry-graphql-cache

from strawberry.extensions import CacheExtension

# Add to schema
schema = strawberry.Schema(
    query=Query,
    mutation=Mutation,
    extensions=[
        CacheExtension(cache_ttl=60),  # 60 second TTL
    ],
)

# Use @strawberry.field(cache=True) for expensive fields
@strawberry.type
class User:
    id: strawberry.ID
    name: str

    @strawberry.field(cache_control=CacheControl(max_age=300))
    async def posts(self, info: Info) -> List[Post]:
        """Cached for 5 minutes."""
        return await fetch_user_posts(self.id)

HTTP response caching with Redis:

# middleware/response_cache.py
import hashlib
import json
from fastapi import Request, Response
import redis.asyncio as redis

redis_client = redis.from_url("redis://localhost:6379")

async def cache_graphql_response(request: Request, call_next):
    """
    Cache GraphQL responses for idempotent queries.
    Only cache queries (not mutations).
    """
    body = await request.body()

    try:
        graphql_request = json.loads(body)
        operation_type = graphql_request.get("operationName", "")

        # Only cache queries
        if not operation_type.startswith("mutation"):
            # Create cache key from query + variables
            cache_key = hashlib.sha256(
                json.dumps(graphql_request, sort_keys=True).encode()
            ).hexdigest()

            # Check cache
            cached = await redis_client.get(f"graphql:{cache_key}")
            if cached:
                return Response(
                    content=cached,
                    media_type="application/json",
                    headers={"X-Cache": "HIT"},
                )

            # Execute request
            response = await call_next(request)

            # Cache successful responses
            if response.status_code == 200:
                await redis_client.setex(
                    f"graphql:{cache_key}",
                    300,  # 5 minute TTL
                    response.body,
                )
                response.headers["X-Cache"] = "MISS"

            return response
    except Exception:
        pass  # Fall through to normal execution

    return await call_next(request)

5. Monitoring and Observability¶

Structured Logging¶

Configure production logging:

# logging_config.py
from loguru import logger
import sys
import json

# Remove default handler
logger.remove()

# Add JSON structured logging for production
logger.add(
    sys.stdout,
    format="{message}",
    serialize=True,  # JSON output
    level="INFO",
)

# Log GraphQL operations
@app.middleware("http")
async def log_graphql_operations(request: Request, call_next):
    if request.url.path == "/graphql":
        body = await request.body()

        try:
            graphql_request = json.loads(body)
            operation_name = graphql_request.get("operationName")

            logger.info(
                "GraphQL request",
                extra={
                    "operation": operation_name,
                    "user_id": request.state.user_id if hasattr(request.state, "user_id") else None,
                    "ip": request.client.host,
                },
            )

            start_time = time.time()
            response = await call_next(request)
            duration = time.time() - start_time

            logger.info(
                "GraphQL response",
                extra={
                    "operation": operation_name,
                    "duration_ms": round(duration * 1000, 2),
                    "status_code": response.status_code,
                },
            )

            return response
        except Exception as e:
            logger.error(f"GraphQL logging error: {e}")

    return await call_next(request)

Instrument GraphQL operations:

# Add to middleware
@app.middleware("http")
async def track_graphql_metrics(request: Request, call_next):
    if request.url.path == "/graphql":
        body = await request.body()
        graphql_request = json.loads(body)

        operation = graphql_request.get("operationName", "unknown")
        operation_type = "mutation" if "mutation" in graphql_request.get("query", "").lower() else "query"

        # Track request
        graphql_requests_total.labels(
            operation=operation,
            operation_type=operation_type,
        ).inc()

        # Track duration
        with graphql_request_duration_seconds.labels(
            operation=operation,
            operation_type=operation_type,
        ).time():
            response = await call_next(request)

        return response

    return await call_next(request)

Health Checks¶

Implement comprehensive health checks:

# health.py
from fastapi import APIRouter
from sqlalchemy import text

router = APIRouter()

@router.get("/health")
async def health_check():
    """Basic liveness check."""
    return {"status": "healthy"}

@router.get("/health/ready")
async def readiness_check(db: AsyncSession = Depends(get_db)):
    """
    Readiness check - verify dependencies are available.
    Used by Kubernetes readiness probe.
    """
    checks = {}

    # Database check
    try:
        await db.execute(text("SELECT 1"))
        checks["database"] = "healthy"
    except Exception as e:
        checks["database"] = f"unhealthy: {str(e)}"

    # Redis check
    try:
        await redis_client.ping()
        checks["redis"] = "healthy"
    except Exception as e:
        checks["redis"] = f"unhealthy: {str(e)}"

    # Overall status
    is_healthy = all(v == "healthy" for v in checks.values())
    status_code = 200 if is_healthy else 503

    return JSONResponse(
        content={
            "status": "healthy" if is_healthy else "unhealthy",
            "checks": checks,
        },
        status_code=status_code,
    )

Migration Scenarios¶

REST to GraphQL Migration¶

Incremental migration strategy:

Phase 1: Parallel Operation (Months 1-3)
├── Maintain REST endpoints
├── Add GraphQL alongside REST
├── Gradually migrate clients
└── Monitor usage metrics

Phase 2: GraphQL Primary (Months 4-6)
├── Encourage GraphQL adoption
├── Deprecate REST endpoints
├── Redirect traffic
└── Update documentation

Phase 3: REST Sunset (Months 7-9)
├── Remove deprecated endpoints
├── Clean up legacy code
└── Full GraphQL adoption

Implementation approach:

# Step 1: Wrap existing REST logic in GraphQL resolvers
@strawberry.type
class Query:
    @strawberry.field
    async def user(self, id: strawberry.ID) -> User:
        """
        Initially delegates to existing REST service.
        Later refactor to direct database access.
        """
        # Call existing REST service
        from src.api.services.user_service import get_user
        user_data = await get_user(id)

        # Convert to GraphQL type
        return User(**user_data)

# Step 2: Add GraphQL-specific optimizations
@strawberry.type
class User:
    id: strawberry.ID
    name: str

    @strawberry.field
    async def posts(self, info: Info) -> List[Post]:
        """
        Use DataLoader for efficient batching.
        This optimization not possible in REST.
        """
        return await info.context.dataloaders.posts_by_author_loader.load(self.id)

# Step 3: Deprecate REST endpoints
@app.get("/api/users/{user_id}")
@deprecated("Use GraphQL query 'user(id: ID)' instead")
async def get_user_rest(user_id: str):
    """Deprecated: Migrate to GraphQL."""
    warnings.warn("REST endpoint deprecated, use GraphQL", DeprecationWarning)
    return {"message": "Please use GraphQL endpoint"}

Client migration:

// Before: REST API calls
const user = await fetch('/api/users/123').then(r => r.json())
const posts = await fetch('/api/posts?author=123').then(r => r.json())

// After: Single GraphQL query
const { user } = await graphqlClient.query({
  query: gql`
    query GetUserWithPosts($id: ID!) {
      user(id: $id) {
        id
        name
        posts {
          id
          title
        }
      }
    }
  `,
  variables: { id: '123' },
})

Schema Evolution¶

Non-breaking changes (safe):

# ✅ Add new fields
type User {
  id: ID!
  name: String!
  bio: String  # New optional field - safe
}

# ✅ Add new types
type Comment {
  id: ID!
  body: String!
}

# ✅ Add new queries/mutations
type Query {
  user(id: ID!): User
  comment(id: ID!): Comment  # New query - safe
}

# ✅ Add optional arguments
type Query {
  users(
    first: Int
    sortBy: String  # New optional argument - safe
  ): [User!]!
}

Breaking changes (requires migration):

# ❌ Remove fields (breaking)
type User {
  id: ID!
  # email: String!  # Removed - BREAKING
}

# ❌ Change field types (breaking)
type User {
  id: ID!  # Was: String! - BREAKING
}

# ❌ Make optional field required (breaking)
type User {
  id: ID!
  email: String!  # Was: String - BREAKING
}

# ❌ Remove arguments (breaking)
type Query {
  users: [User!]!  # Removed 'first' argument - BREAKING
}

Deprecation strategy:

# Mark fields as deprecated before removal
@strawberry.type
class User:
    id: strawberry.ID
    name: str

    email: str = strawberry.field(
        deprecation_reason="Use 'emailAddress' instead. Will be removed in v2.0"
    )

    email_address: str  # New field to replace 'email'

# Schema introspection shows deprecation:
# {
#   "name": "email",
#   "isDeprecated": true,
#   "deprecationReason": "Use 'emailAddress' instead..."
# }

# Migration timeline:
# - v1.5: Add new field, deprecate old field
# - v1.6-1.9: Both fields available, warnings logged
# - v2.0: Remove deprecated field (breaking change)

Versioning strategies:

# Strategy 1: Field-level versioning (recommended)
@strawberry.type
class User:
    id: strawberry.ID
    name_v1: str = strawberry.field(
        deprecation_reason="Use 'fullName' instead"
    )
    full_name: str  # New version

# Strategy 2: Schema versioning (GraphQL endpoint paths)
# /graphql/v1 - Old schema
# /graphql/v2 - New schema with breaking changes

# Strategy 3: Custom directives
@strawberry.type
class User:
    @strawberry.field(
        directives=[deprecated(version="2.0", reason="Use fullName")]
    )
    def name(self) -> str:
        return self.full_name

Database Migration with GraphQL¶

Coordinate database and schema changes:

# Migration 001: Add 'bio' column to users table
# database/migrations/001_add_user_bio.py

from alembic import op
import sqlalchemy as sa

def upgrade():
    # Add column to database
    op.add_column('users', sa.Column('bio', sa.Text(), nullable=True))

def downgrade():
    op.drop_column('users', 'bio')

# GraphQL schema change (deployed simultaneously)
@strawberry.type
class User:
    id: strawberry.ID
    name: str
    bio: Optional[str] = None  # New field matches database column

# Deployment order:
# 1. Deploy database migration (add column)
# 2. Deploy GraphQL schema update (add field)
# 3. Clients can now query new field

Handle backward compatibility during rollout:

# Support both old and new schemas during migration
@strawberry.type
class User:
    id: strawberry.ID

    # Old field (deprecated)
    @strawberry.field(deprecation_reason="Use 'fullName' instead")
    def name(self) -> str:
        # Map to new database column
        return self.full_name

    # New field
    full_name: str  # Maps to 'full_name' column in database

# Database migration:
# 1. Add 'full_name' column
# 2. Backfill from 'name' column
# 3. Keep both columns during transition
# 4. Remove 'name' column after full migration

Troubleshooting Guide¶

Common Issues¶

Issue: “Using default JWT secret key” Warning¶

Symptom:

UserWarning: Using default JWT secret key. Set RISO_GRAPHQL_JWT_SECRET_KEY environment variable in production.

Solution:

# Generate a secure secret key
python3 -c "import secrets; print(secrets.token_urlsafe(64))"

# Set environment variable
export RISO_GRAPHQL_JWT_SECRET_KEY="<generated-key>"

# Verify configuration
uv run python -c "from your_package.graphql_api.config import get_auth_settings; print(get_auth_settings().jwt_secret_key)"

Issue: “Query complexity exceeded” Errors¶

Symptom:

{
  "errors": [{
    "message": "Query complexity 7500 exceeds maximum 5000. Please reduce the number of fields or list sizes requested."
  }]
}

Solution:

# Option 1: Increase limit (if query is legitimate)
export RISO_GRAPHQL_MAX_QUERY_COMPLEXITY=10000

# Option 2: Optimize query (reduce requested items)
# Before:
query {
  users(first: 100) {
    posts(first: 100) { title }  # 100 * 100 = 10,000 complexity
  }
}

# After:
query {
  users(first: 20) {
    posts(first: 10) { title }   # 20 * 10 = 200 complexity
  }
}

Issue: N+1 Query Problems¶

Symptom: Database logs show many individual SELECT queries:

SELECT * FROM users WHERE id = 1;
SELECT * FROM posts WHERE author_id = 1;
SELECT * FROM users WHERE id = 2;
SELECT * FROM posts WHERE author_id = 2;
-- ... repeated N times

Solution:

# Use DataLoaders (already implemented in template)

# Verify DataLoaders are being used:
@strawberry.type
class User:
    @strawberry.field
    async def posts(self, info: Info) -> List[Post]:
        # ✅ Uses DataLoader (batching)
        return await info.context.dataloaders.posts_by_author_loader.load(self.id)

        # ❌ Direct query (N+1 problem)
        # return await fetch_posts_for_user(self.id)

# Monitor DataLoader effectiveness:
# - Check batch sizes in logs
# - Expected: 1-2 queries per request
# - Actual N+1: N+1 queries per request

Issue: Token Validation Failures¶

Symptom:

{
  "errors": [{
    "message": "Invalid token signature",
    "extensions": { "code": "AUTH_ERROR" }
  }]
}

Diagnosis checklist:

# Check 1: Verify secret key matches token generation
# Token must be signed with same key used for verification

# Check 2: Algorithm mismatch
# Generation: HS256, Verification: HS512 -> FAIL
export RISO_GRAPHQL_JWT_ALGORITHM="HS256"  # Must match

# Check 3: Token expiration
# Verify system clocks are synchronized (NTP)

# Check 4: Issuer/audience mismatch
# Token: {"iss": "https://api.example.com"}
# Config: RISO_GRAPHQL_JWT_ISSUER="https://api.other.com"  # FAIL

# Debug token contents:
import jwt
token = "eyJhbGci..."
payload = jwt.decode(token, options={"verify_signature": False})
print(payload)  # Inspect claims

Issue: High Memory Usage¶

Symptom: Application memory grows over time, eventual OOM (Out of Memory).

Diagnosis:

# Check 1: Connection pool leaks
# Verify connections are being returned to pool

# Check 2: DataLoader cache size
# DataLoaders cache per-request, but ensure requests are completing

# Check 3: Database session leaks
# Always use dependency injection for database sessions
@app.post("/graphql")
async def graphql_endpoint(db: AsyncSession = Depends(get_db)):
    # Session automatically closed after request
    ...

# Check 4: Large response payloads
# Monitor response sizes in logs
# Implement pagination for large lists

Solutions:

# Configure connection pool limits
engine = create_async_engine(
    DATABASE_URL,
    pool_size=20,
    max_overflow=10,
    pool_recycle=3600,  # Recycle connections hourly
)

# Limit pagination sizes
export RISO_GRAPHQL_MAX_PAGE_SIZE=50  # Reduce from 100

# Enable response compression
from fastapi.middleware.gzip import GZipMiddleware
app.add_middleware(GZipMiddleware, minimum_size=1000)

Issue: Slow Query Performance¶

Symptom: GraphQL queries taking >1 second to complete.

Diagnosis tools:

# Enable query timing logs
@app.middleware("http")
async def log_query_timing(request: Request, call_next):
    start = time.time()
    response = await call_next(request)
    duration = time.time() - start

    if duration > 1.0:  # Log slow queries
        logger.warning(f"Slow query: {duration:.2f}s")

    return response

# Profile database queries
engine = create_async_engine(
    DATABASE_URL,
    echo=True,  # Enable SQL logging temporarily
)

# Identify missing indexes

Production Checklist¶

Use this checklist before going live:

Security¶

  • JWT secret key generated with cryptographic random source

  • Secret stored in encrypted vault (not in code/environment files)

  • Algorithm configured correctly (HS256 for single-server, RS256 for distributed)

  • Token expiration set (15-30 minutes recommended)

  • Issuer and audience claims configured

  • Introspection disabled (ENABLE_INTROSPECTION=false)

  • GraphQL Playground disabled (ENABLE_PLAYGROUND=false)

  • CORS configured with specific origins (not *)

  • Rate limiting implemented and tested

  • Query depth limit set (5-10 for public APIs)

  • Query complexity limit set (3000-5000 for public APIs)

  • HTTPS/TLS configured on load balancer/ingress

  • Security headers configured (CSP, HSTS, etc.)
