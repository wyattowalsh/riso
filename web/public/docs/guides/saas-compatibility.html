Migration Paths¶

Database Migration¶

Supabase → Neon¶

Reason: Better edge performance, lower cost at scale.

# 1. Export Supabase database
pg_dump $SUPABASE_URL > supabase_dump.sql

# 2. Create Neon database
neon projects create --name my-app

# 3. Import to Neon
psql $NEON_URL < supabase_dump.sql

# 4. Migrate storage (Supabase Storage → R2)
node scripts/migrate-storage-to-r2.js

# 5. Update environment variables
# DATABASE_URL=$NEON_URL
# Remove SUPABASE_* vars

Downtime: ~15 minutes (for read-only mode + cutover)

Gotchas:

  • Supabase Auth → Clerk migration requires user data export

  • Realtime subscriptions need alternative (Inngest, websockets)

  • Storage migration is one-way (R2 doesn’t support Supabase transforms)

Neon → Supabase¶

Reason: Need realtime features, storage integration.

# 1. Create Supabase project
supabase projects create my-app

# 2. Export Neon database
pg_dump $NEON_URL > neon_dump.sql

# 3. Import to Supabase
psql $SUPABASE_URL < neon_dump.sql

# 4. Enable Supabase features
supabase db push
supabase storage create-bucket uploads

ORM Migration¶

Prisma → Drizzle¶

Reason: Better edge compatibility, smaller bundle size.

Step 1: Install Drizzle

pnpm add drizzle-orm @neondatabase/serverless
pnpm add -D drizzle-kit

Step 2: Convert Schema

// Before (Prisma)
model User {
  id        String   @id @default(cuid())
  email     String   @unique
  name      String?
  createdAt DateTime @default(now())
}

// After (Drizzle)
import { pgTable, text, timestamp } from 'drizzle-orm/pg-core';

export const users = pgTable('users', {
  id: text('id').primaryKey().$defaultFn(() => createId()),
  email: text('email').notNull().unique(),
  name: text('name'),
  createdAt: timestamp('created_at').notNull().defaultNow(),
});

Step 3: Migrate Queries

// Before (Prisma)
const user = await prisma.user.findUnique({
  where: { email: 'user@example.com' },
});

// After (Drizzle)
const user = await db.query.users.findFirst({
  where: eq(users.email, 'user@example.com'),
});

Migration Time: 2-8 hours (depending on schema complexity)

Automation Script:

# Convert Prisma schema to Drizzle
npx prisma-to-drizzle schema.prisma > drizzle.schema.ts

Drizzle → Prisma¶

Reason: Need Prisma Client features (middleware, metrics).

Similar process in reverse, use drizzle-to-prisma converter.

Auth Migration¶

AuthJS → Clerk¶

Reason: Better UX, pre-built components, org management.

Data Migration:

// 1. Export users from AuthJS
const users = await prisma.user.findMany({
  include: { accounts: true },
});

// 2. Import to Clerk
for (const user of users) {
  await clerkClient.users.createUser({
    emailAddress: [user.email],
    firstName: user.name?.split(' ')[0],
    lastName: user.name?.split(' ')[1],
    publicMetadata: { legacyId: user.id },
  });
}

Downtime: ~1 hour (user re-authentication required)

Storage Migration¶

R2 → Supabase Storage¶

Reason: Need image transformations, simpler auth model.

import { r2, supabase } from '@/lib/clients';

async function migrateFiles() {
  const { files } = await r2.listFiles({ prefix: 'uploads/' });

  for (const file of files) {
    // Download from R2
    const buffer = await r2.downloadFile(file.key);

    // Upload to Supabase
    await supabase.uploadFile({
      path: file.key,
      file: buffer,
      contentType: file.contentType,
    });

    console.log(`Migrated: ${file.key}`);
  }
}

Migration Time: Depends on file count (10K files = ~1 hour)

Cost Optimization Strategies¶

1. Database Costs¶

Neon Autoscaling¶

# neon.toml
[compute]
autoscaling_limit_min_cu = 0.25  # Sleep to zero
autoscaling_limit_max_cu = 2     # Cap at 2 CU
autosuspend_delay_seconds = 300  # 5 min idle timeout

Savings: ~70% reduction vs always-on instance

Connection Pooling¶

// Use Neon serverless driver (HTTP-based)
import { neon } from '@neondatabase/serverless';

const sql = neon(process.env.DATABASE_URL!);

// Or use connection pooling
const pooledUrl = process.env.DATABASE_URL!.replace(
  'postgres://',
  'postgres-pooler://'
);

Impact: Reduce connection overhead by 90%

Query Optimization¶

// ❌ N+1 query problem
const users = await db.query.users.findMany();
for (const user of users) {
  const posts = await db.query.posts.findMany({
    where: eq(posts.userId, user.id),
  });
}

// ✅ Use joins
const users = await db.query.users.findMany({
  with: { posts: true },
});

Impact: 100x faster, 10x cheaper

Troubleshooting¶

Common Issues¶

1. Prisma on Cloudflare Workers¶

Error: PrismaClient is unable to run in Cloudflare Workers

Solution: Use Drizzle ORM or Prisma Data Proxy

// Option A: Switch to Drizzle (recommended)
import { drizzle } from 'drizzle-orm/neon-http';
const db = drizzle(process.env.DATABASE_URL);

// Option B: Use Prisma Data Proxy (adds cost + latency)
// DATABASE_URL=prisma://aws-us-east-1.prisma-data.com/?api_key=xxx

2. Neon + Supabase Storage¶

Error: Supabase Storage requires Supabase Database

Solution: Choose one of:

  • Full Supabase: database: supabase + storage: supabase-storage

  • Neon + R2: database: neon + storage: r2

3. Clerk + Supabase Auth Conflict¶

Warning: Both Clerk and Supabase provide authentication

Solution: Disable Supabase Auth in dashboard:

  1. Go to Supabase Dashboard → Authentication

  2. Disable “Enable email confirmations”

  3. Use Clerk as single source of truth for auth

4. High R2 Egress from Vercel¶

Issue: R2 egress is free within Cloudflare network, but costs apply from Vercel

Solutions:

  1. Use Cloudflare CDN in front of R2

  2. Switch to Vercel Blob (if budget allows)

  3. Accept egress costs (still cheaper than S3)

5. Background Jobs Not Running¶
